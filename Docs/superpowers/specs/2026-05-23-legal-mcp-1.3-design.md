# Legal-MCP 1.3 Design

## Summary

Legal-MCP 1.3 is a breaking security release. It replaces broad project-context
queries with a minimum-disclosure query architecture: identify the user's
intended data domain and fields, authorize that exact scope, call fine-grained
tools, and return only the smallest necessary result.

The 1.2 changes that scoped named `legal` users to project grants and introduced
project field filtering are included in 1.3 as transitional work. They are not
the final model. In 1.3, unrestricted full-context responses are not allowed for
legacy or new query paths.

## Goals

- Prohibit default full project-context disclosure.
- Split MCP query tools by data domain and field-level capability.
- Provide a machine-readable tool catalog for client-side and server-side
  planning.
- Add a server-side query planner agent that decides which registered
  fine-grained tools to call for clients that want a single planned-query entry
  point.
- Enforce role plus user-group permissions across project, data domain, field,
  and operation dimensions.
- Extend the admin UI to manage user groups, granular permissions, project
  aliases, and import review workflows.
- Import "contract information" CSV/XLSX files by matching each contract to a
  project through aliases first, exact project lookup second, and fuzzy
  candidates only with user confirmation.
- Add MCP startup hooks for local compatibility checks and non-blocking remote
  version checks.

## Non-Goals

- Do not keep `get_project_context` as a full-context public query path.
- Do not rely on prompt instructions to prevent over-disclosure.
- Do not let the planner query the database directly.
- Do not auto-import contracts from fuzzy project matches.
- Do not make remote update checks required for server startup.

## Core Disclosure Rule

All MCP answers follow this flow:

1. Parse the user question into data domains, entity references, filters, and
   requested fields.
2. Resolve referenced entities, such as project aliases or contract numbers.
3. Authorize the user's role and user groups for the requested operation,
   project scope, data domain, and fields.
4. Call only registered fine-grained tools.
5. Return only permitted identifiers plus explicitly requested fields.
6. Run a final disclosure check before serializing the answer.

If the system cannot determine the minimum necessary fields, it must ask for
clarification or return a validation error. It must not fall back to a full
context response.

## Tool Architecture

### Tool Catalog

Every MCP tool must be registered in a machine-readable catalog. The catalog is
part of the product surface, not just documentation. Each entry includes:

- Tool name and human description.
- Data domain: project, contract, party, license, asset, risk, audit, admin.
- Supported filters.
- Supported return fields.
- Required operation permissions.
- Whether project scope is required.
- Whether the tool returns a list or a single record.
- Default limit, maximum limit, and sorting behavior for list tools.
- Disclosure classification for each field.
- Examples of user questions that map to the tool.

Clients can inspect the catalog and plan tool calls themselves. The server-side
planner also uses the same catalog.

### Fine-Grained Tool Families

1.3 starts by replacing broad project context with small tools. Initial tool
families include:

- Project tools:
  - `resolve_project`
  - `get_project_fields`
  - `list_projects`
- Contract tools:
  - `resolve_contract`
  - `list_project_contracts`
  - `get_contract_fields`
- Party tools:
  - `resolve_party`
  - `list_project_parties`
  - `get_party_fields`
- License tools:
  - `list_project_licenses`
  - `get_license_fields`
- Risk tools:
  - `list_project_risks`
  - `get_risk_fields`
- Asset tools, added when the asset schema lands:
  - `list_project_assets`
  - `get_asset_fields`
- Audit and admin tools:
  - only visible to groups with explicit audit/admin operations.

This list is intentionally extensible. Adding domains such as subjects, licenses,
assets, and future contract subtypes requires registering new fine-grained
tools and catalog entries, not expanding one large context tool.

### Deprecated Tool Behavior

`get_project_context` is deprecated in 1.3. If it remains visible temporarily,
it must reject calls that do not provide an explicit field intent. It must not
return `projects.*`, `licenses.*`, `contracts.*`, and `risks.*` as one response.

## Planner Agent

1.3 supports both client-side and server-side planning.

Client-side planning:

- MCP exposes the tool catalog plus fine-grained tools.
- Capable clients, such as Codex or Claude, choose tools from the catalog.
- The server still enforces policy on every tool call.

Server-side planning:

- MCP exposes a planner entry point, such as `plan_query` or
  `answer_query`.
- The planner receives the user question and context.
- It inspects the catalog and calls registered tools.
- It cannot query SQLite directly.
- It must include the planned data domains, fields, tools, and permissions in
  an auditable plan record.
- Final answers are generated only from allowed tool results.

The planner's output includes:

- Recognized entities.
- Requested fields.
- Candidate tools.
- Authorized fields.
- Denied fields and reasons.
- Final result shape.

## Permission Model

1.3 uses a two-layer model:

- Role: `admin`, `legal`, `business`, `auditor`.
- User groups: custom groups such as "ACME Project Team", "Publishing", or
  "Legal Outsourcing".

Roles define baseline capability ceilings. User groups grant concrete access
within those ceilings. A user can belong to multiple groups.

Permission dimensions:

- Project scope.
- Data domain scope.
- Field scope.
- Operation scope:
  - `read`
  - `import`
  - `manage_users`
  - `manage_keys`
  - `manage_permissions`
  - `view_audit`

All query tools, import commands, and admin handlers must use the same policy
layer. Business logic must not independently decide visibility.

## Data Model Changes

New or expanded tables:

- `user_groups`
- `user_group_memberships`
- `permission_grants`
- `project_aliases`
- `schema_version`
- `import_batches`
- `import_review_items`

`permission_grants` stores user-group permissions across operation, project,
data domain, and field dimensions. The schema supports wildcard-like
grants for admin convenience, but policy evaluation must resolve them into
explicit allowed fields before data is returned.

`project_aliases` maps external names such as `ACME项目部` to canonical projects.
Aliases are managed in the admin UI and used by import and query resolution.

The existing `contracts` table needs more columns for contract information
imports:

- handler
- payment_terms
- currency
- total_amount
- expiry_date
- signed_date
- counterparty
- company_entity
- title
- contract_number
- income_expense_type
- summary

Existing `external_key` uses the contract number when available.

## Admin UI

The admin UI adds:

- User group creation and deletion.
- User group membership management.
- Project-scope grants.
- Data-domain and field-level grants.
- Operation grants.
- Project alias management.
- Contract import review for unresolved and fuzzy project matches.
- Audit visibility controlled by permission grants.

Project authorization changes from "can this user see this project" to "which
groups can perform which operations on which fields in this project and domain."

## Contract Information Import

The import pipeline recognizes a "contract information" table when headers
include contract-specific fields such as:

- 项目代号
- 经办人
- 收付款条件
- 合同概览
- 币种
- 总金额
- 过期时间
- 签约时间
- 对方签约公司
- 我方签约公司
- 合同主题
- 合同号
- 收入或支出

Project matching order:

1. Project alias match, for example `ACME项目部` -> `ACME`.
2. Exact project code or project name match.
3. Fuzzy candidates returned for user confirmation.

Only alias and exact matches can write contracts directly into the canonical
`contracts` table. Fuzzy and unresolved rows go into an import review workflow.
The review page shows the raw project value, candidate projects, confidence
metadata, and the contract fields that would be imported after confirmation.

## Startup Hooks

Both stdio MCP and HTTP MCP execute startup checks.

Local checks:

- Current app version.
- Database schema version.
- Migration compatibility.
- Required tables and columns for the current app version.

If local compatibility fails, startup fails with a clear error. This
prevents old schemas from serving new fine-grained tools incorrectly.

Remote checks:

- Optional PyPI, GitHub release, or configured URL check.
- Short timeout.
- Disabled by configuration when needed.
- Network failure never blocks startup.
- Successful checks log or expose update notices.

## Auditing

Audit events must capture:

- User and API key.
- Role and matched user groups.
- Planner decision, when a planner is used.
- Tool calls.
- Requested fields.
- Returned fields.
- Denied fields and reasons.
- Project and record identifiers.
- Import match mode: alias, exact, fuzzy_candidate, unresolved.
- Result count and error code.

The audit system supports proving that a query returned only the minimal
allowed fields.

## Migration and Compatibility

1.3 is a breaking upgrade from 1.2.

- `get_project_context` no longer returns full context.
- Tools that previously returned whole rows must move to explicit field lists.
- Named `legal` users remain scoped by project and group permissions.
- Existing project grants are migrated into default user groups or
  equivalent permission grants.
- Documentation must state the 1.2 -> 1.3 behavior change clearly.

The migration preserves existing data and API keys where possible, but
query behavior intentionally changes for safety.

## Testing Strategy

Unit tests:

- Policy evaluation for roles, groups, operations, projects, domains, and fields.
- Tool catalog validation.
- Fine-grained tool field allowlists.
- Planner output for representative user questions.
- Contract import header detection and field normalization.
- Alias, exact, fuzzy, and unresolved project matching.
- Startup hook local compatibility checks.

Integration tests:

- MCP tools/list includes catalog metadata.
- Deprecated `get_project_context` rejects full-context calls.
- Querying "official website" returns only project identifiers and website.
- Querying a contract amount returns only contract identifiers and amount fields.
- A group without contract field access cannot retrieve contract summaries.
- Contract CSV import writes alias/exact matches and creates review items for
  fuzzy candidates.
- HTTP and stdio startup paths both execute local checks.

Regression tests:

- No public tool response can include `select *` style full rows unless the
  requested and authorized field set explicitly covers every returned field.
- Remote version check failures do not block startup.

## Rollout Plan

1. Add schema versioning and startup compatibility checks.
2. Add user-group and permission schema.
3. Migrate existing project access grants into group-based grants.
4. Build the policy evaluation layer.
5. Add the tool catalog.
6. Add fine-grained project tools and deprecate full project context.
7. Add contract schema expansion and contract information import.
8. Add project aliases and import review workflow.
9. Add planner support using the tool catalog.
10. Extend admin UI for user groups, permissions, aliases, and review.
11. Update documentation for the 1.3 breaking security upgrade.

## Acceptance Criteria

- No MCP query can return full project context by default.
- Every returned field has a policy decision explaining why it was allowed.
- The tool catalog describes all exposed fine-grained tools.
- The planner can choose tools from the catalog but cannot bypass policy.
- Admins can configure user groups, project scope, field scope, and operation
  scope.
- Contract information imports match projects by alias or exact lookup, and
  fuzzy matches require confirmation.
- Startup hooks run for stdio and HTTP transports.
- Full test suite passes with regression coverage for minimum disclosure.
