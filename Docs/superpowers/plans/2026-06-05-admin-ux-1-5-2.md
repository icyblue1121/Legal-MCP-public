# Legal-MCP 1.5.2 开发方案（管理界面细微改动）

> **For agentic workers:** REQUIRED SUB-SKILL: 用 superpowers:subagent-driven-development 或 superpowers:executing-plans 逐任务实现。任务用 `- [ ]` 复选框跟踪。

> **范围说明：** 原 1.5.2 里属于**数据库重构级别**的大改（迁移框架、主体建表、批量写、导入落库、后台变更审计）已上移为 [1.6.0 开发方案](2026-06-05-database-refactor-1-6-0.md)。本 1.5.2 只做**管理界面的细微改动**，不改既有表结构、不引入新依赖、不动权限模型。（唯一的数据持久化变更是**新增一张审计详情 sidecar 表**——新表在现有 `create table if not exists` 机制下对老库也安全，不依赖 1.6.0 的迁移框架，见 Phase 3 说明。）

> **审计详情数据源已定：混合方案。** agent 类事件（`agent_query`/`structured_query`）**深链到其 Langfuse trace**（在 Phase 4 的内嵌 Langfuse 里打开），只在 sidecar 存 `thread_id` 关联键、不重复存全文；非 agent 事件（直连工具、无 LLM trace）把其（通常较小的）`arguments`/`result` 落进同一张 sidecar 表。需把 `thread_id` 打通进审计写入路径（管线小改）。

**背景：** 1.5.1 已落地 Users 重构、管理页（Tab/分页/搜索/排序/保持位置）、Database 页，并把 `admin_server.py` 按 Mixin 拆成解耦模块（`admin_render/users/manage/database/misc/common`）。1.5.2 在此之上打磨四个小而具体的体验点。

**本期目标（四项）：**
1. **部署模式切换开关**：在后台界面把"个人部署（local）↔ 团队部署（team）"做成可切换的开关。**护栏：切到 team 必须先设置 admin 密码并验证登录成功，才允许切换**，避免切过去后没密码把自己锁死（死循环）。
2. **审计页全量分页**：每页 100 条、可翻页浏览**全部**审计记录（现状只显示最近 100 条、无翻页）。
3. **审计条目详情 + Record 列正名**：每条审计可**点开详情**，看到**服务器返回的全部内容、问题原文、回答原文**（文字多时用详情弹窗/抽屉承载）；同时把含义不清的 "Records" 列正名。
4. **接入 Langfuse 管理页**：把已自托管的 Langfuse 后台接进管理界面，**从 Audit 页可直接进入**，且**不出现 Langfuse 自己的登录页**（管理后台本身已鉴权）。

**架构基线：** 延续 1.5.1 的轻量 `http.server` + Mixin 结构。不引入前端框架。

## Langfuse 免登录可行性研究（结论先行）

> 用户要求"研究一下怎样才能实现"。已查官方文档与社区，结论如下。

**1. Langfuse 没有"完全关闭鉴权"的开关。** 平台始终需要一个会话；`AUTH_DISABLE_USERNAME_PASSWORD=true` 只是关掉账号密码、强制改用 SSO，并不能免鉴权（[官方 SSO 文档](https://langfuse.com/self-hosting/security/authentication-and-sso)、[Discussion #4212](https://github.com/orgs/langfuse/discussions/4212)）。所以"不要登录页"只能靠**我们替用户把会话准备好**。

**2. 推荐实现：同源反向代理 + 服务端持有会话、向上游注入 Cookie。**
- 管理服务器把 `/admin/observability/*` 反代到 Langfuse；**浏览器只跟管理后台同源通信**，是否带 Langfuse 的 Cookie 无所谓——**Cookie 由代理在"代理→Langfuse"这一跳注入**。于是跨域 Cookie 限制不构成问题，登录页也永不出现。
- 会话怎么来：管理服务器用已配置的 `LANGFUSE_INIT_USER_EMAIL/PASSWORD` 走 NextAuth 的 credentials 登录流程（`GET /api/auth/csrf` → `POST /api/auth/callback/credentials`），拿到 `next-auth.session-token` Cookie，缓存复用、过期重登。访问入口本身已被管理后台鉴权保护。

**3. 唯一硬约束：子路径托管要重建镜像。** Langfuse 把 base path **内联进静态资源**，要在 `/admin/observability` 子路径下跑，必须用 `NEXT_PUBLIC_BASE_PATH` **从源码重新构建镜像**（预构建的 `ghcr.io/langfuse/langfuse:3` 不支持任意子路径）——见[官方 Custom Base Path](https://langfuse.com/self-hosting/configuration/custom-base-path)。因此 Phase 4 要在两条路里**做一次 spike 选型**：
   - **(A) 重建镜像**：`NEXT_PUBLIC_BASE_PATH=/admin/observability` + `NEXTAUTH_URL=…/admin/observability/api/auth`，干净地挂在管理后台子路径下。代价：维护一个自构建镜像。
   - **(B) 不重建、代理到"根路径"**：管理侧用一个**子监听器/独立端口**把 Langfuse 代理在其根路径（无需 base path），仍服务端注入会话 Cookie；入口从 Audit 链过去。Cookie 是按主机（`127.0.0.1`）而非端口隔离，管理会话 Cookie 同主机可达，便于在该监听器上复用管理鉴权。代价：多一个监听器、URL 不在 `/admin/` 之下。

**4. 兜底：** 若 spike 发现注入会话不稳定，退到"iframe 直链 + 文档说明需先在 Langfuse 登录一次"，或留到后续版本用 Langfuse API 自绘视图。

## 假设

- 不改既有表结构；新增 `audit_event_payloads` 表（新表对老库安全）。schema_version 常量可随之 +1，但不依赖迁移执行器。
- Langfuse 仍按现有 compose 自托管在 localhost，凭据由环境变量注入；本期只加"管理端入口 + 免登录代理（+ 可能的自构建镜像）"，不改 Langfuse 鉴权体系本身。
- 反向代理只在**管理后台鉴权通过后**可达，不对公网暴露。

## 非目标

- 不做 schema 迁移框架、主体表、批量写、导入落库、后台变更审计（全在 [1.6.0](2026-06-05-database-refactor-1-6-0.md)）。
- 不改既有表的列、不重写权限模型、不引入前端框架。
- 不重做 Langfuse 部署/鉴权，不自研 trace 可视化。

## 成功标准

- 后台有**部署模式开关**；从 local 切 team 会**强制先设密码并验证登录成功**再翻模式，无法把自己锁死；切换持久化、无需重启即生效，导航栏标签随之更新。
- Audit 页**每页 100 条**且**可翻到全部**历史（首/末/上/下页 + 总数正确）。
- Audit 每条可**点开详情**：agent 事件深链到 Langfuse 看完整问答、非 agent 事件弹窗看完整 JSON；"Records" 列表头正名。
- 从 Audit 页可**一键进入 Langfuse**，过程中**不出现 Langfuse 登录框**；未配置 Langfuse 时入口优雅降级。
- 现有 1.5.1 测试保持全绿；新增行为有针对性测试。

## 发布形态（Phases）

### Phase 1：部署模式切换开关（含防锁死护栏）

- [ ] 模式持久化：在现有设置存储读写 `mode`（缺省取启动参数 `--mode`）。`LegalMCPAdminServer.is_local_mode` 改为读持久化值（每请求读或带失效缓存），启动参数仅作首次缺省。
- [ ] 切换端点：`POST /admin/deployment-mode`，仅已登录 admin 可调。
- [ ] **防锁死护栏（用户要求）**：切到 `team` 走"设密码 → 校验登录成功 → 翻模式"串行流程——若无 admin 密码或验证不通过，**不翻模式**并提示原因；翻到 `local`（去鉴权）需表单二次确认并提示安全含义。
- [ ] UI：导航栏只读 `mode` 标签（[admin_render.py:293](src/legal_mcp/admin_render.py)）处放开关/带确认按钮，切后 flash 结果并刷新标签。
- [ ] 验证：team→local→team 往返持久化、无需重启；无密码切 team 被拦；设密码并登录成功后才翻到 team；local 切换需确认。

### Phase 2：审计页全量分页（每页 100）

- [ ] `list_audit_events` 增加 `offset`（默认 0），并加 `count_audit_events(conn, **filters)`（或返回总数），保持既有过滤参数兼容。
- [ ] `_pager` 增加可选 `page_size`（默认 `_PAGE_SIZE`，不影响现有调用）。新增 `_AUDIT_PAGE_SIZE=100`。
- [ ] 审计页按 `limit=100 offset=(page-1)*100` 取数 + 渲染 `_pager(..., page_size=100)`；处理空结果与越界 page（钳到合法范围）。
- [ ] 验证：构造 >100 条 → 首页 100、可翻到第 2…N 页、总数正确、末页不溢出。

### Phase 3：审计条目详情 + Record 列正名

**3a thread_id 管线打通 + sidecar 持久化（新表，规避迁移依赖）**
- [ ] 把 `thread_id` 从 agent 执行（[agent_graph.py](src/legal_mcp/agent_graph.py)）打通到工具审计写入路径 `_audit_database`/`write_audit_event`（[tools.py:446](src/legal_mcp/tools.py)）——给签名加可选 `thread_id`，agent 路径传入、非 agent 路径为 `None`。
- [ ] 新增 `audit_event_details(audit_event_id PK/FK, thread_id text, arguments_json text, response_json text, truncated int, created_at)`（`create table if not exists`，老库安全）。
- [ ] 在 `write_audit_event` 写主事件后**同事务**写 sidecar：**agent 事件**（有 thread_id）只写 `thread_id`（不自存全文）；**非 agent 事件**写完整 `arguments`/`result`，**设大小上限**（超出截断并标 `truncated`）。
- [ ] schema_version 常量 +1（仅标记，不需迁移执行器）。

**3b 详情视图（混合）**
- [ ] 列表每行加"详情"入口。
- [ ] **agent 事件**：详情里给"在 Langfuse 中查看"按钮，深链到该 trace（用 thread_id = session，跳到 Phase 4 的内嵌 `/admin/observability/...?sessionId=…` 或 trace 页）；Langfuse 未配置时，提示"需启用 Langfuse 查看完整问答"。
- [ ] **非 agent 事件**：用**弹窗/抽屉**展示 sidecar 里的 `arguments`/`result` 完整 JSON（可滚动、超大截断标注）+ 该事件已有 disclosures。
- [ ] 历史事件无 sidecar 行时，详情显示"本次上线前的事件未捕获详情"。
- [ ] 弹窗用最小内联 JS/CSS（不引框架），或退化为 `/admin/audit/<id>` 详情页（见开放决策）。

**3c Record 列正名**
- [ ] 表头 `Records` → "返回记录数 / Records returned"，加一句话说明；理由/查询用直白表头（如"查询/理由"）。
- [ ] 验证：跑一次 `agent_query` → 列表看到返回记录数与查询；点详情 → agent 事件能跳到 Langfuse 看到完整问答、非 agent 事件弹窗看到完整 JSON；表头不再含糊。

### Phase 4：接入 Langfuse 管理页（从 Audit 进入、免登录）

> 先做 **spike**（见上文研究第 3 点）确定子路径(A)/根路径(B)选型，再实现。

- [ ] 在 Audit 页加入口："Observability / Langfuse" 按钮或子标签 → `/admin/observability`；`LANGFUSE_BASE_URL` 等未配置时隐藏或置灰提示"未启用 Langfuse"。
- [ ] 反向代理 `/admin/observability[/*]` → `LANGFUSE_BASE_URL`（容器内 `http://langfuse-web:3000`），仅管理鉴权后可达。
- [ ] 服务端会话：用 `LANGFUSE_INIT_USER_EMAIL/PASSWORD` 走 NextAuth credentials 登录（`/api/auth/csrf` → `/api/auth/callback/credentials`）拿 `next-auth.session-token`，缓存复用、过期重登；代理时向上游注入该 Cookie，使内嵌页面**不出现登录框**。
- [ ] 选型落地：(A) 自构建镜像设 `NEXT_PUBLIC_BASE_PATH=/admin/observability` + 对应 `NEXTAUTH_URL`；或 (B) 子监听器把 Langfuse 代理在根路径。处理 `_next/`、静态资源、`/api/*` 等子资源路径。
- [ ] 兜底：会话注入不稳定则退 iframe 直链 + 文档说明，或留后续自绘视图。
- [ ] 验证：配置好 Langfuse 后从 Audit 点入 → 直接见 trace 界面、**无登录提示**；未配置时入口降级、其余页面不受影响。

## 风险与权衡

- **模式切换防锁死**：去掉 team 鉴权或切到没密码的 team 都可能锁死——护栏（先设密码+验证登录、切 local 二次确认）是本期最敏感点，必须先写测试覆盖。
- **审计详情（混合）的耦合与覆盖**：agent 详情依赖 Phase 4 的内嵌 Langfuse + thread_id 关联键打通（管线小改，需测 agent/非 agent 两路）；Langfuse 未配置/过期时 agent 详情需优雅降级。非 agent 自存全文要设大小上限、明确明文存于本地库的信任边界；历史事件无详情属预期。
- **Langfuse 免登录代理**：最高实现不确定性。需先 spike 验证 NextAuth credentials 会话注入与子路径/根路径选型；做不到走兜底。代理仅管理鉴权后可达，勿暴露公网。
- **`_pager` 改动面**：加 `page_size` 要对现有调用零影响。

## 依赖与顺序

注意：混合方案让 **Phase 3 的 agent 详情依赖 Phase 4 的内嵌 Langfuse**（深链落点）。建议：Phase 2（分页）→ Phase 4（Langfuse spike + 内嵌，给详情深链提供落点）→ Phase 3（thread_id 打通 + sidecar + 详情/正名）→ Phase 1（模式开关）。Phase 3 的非 agent 详情与列正名不依赖 Phase 4，可并行先做。
