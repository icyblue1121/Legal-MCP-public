# Legal-MCP 战略转向报告：从「押注未来的商业平台」到「现在就能用的开源治理层」

> 生成日期：2026-06-06
> 性质：战略定位报告，由一次关于「产品本意 vs 开发路线是否矛盾」的讨论整理而来。
> 状态：待决策（见 §6 第 5 项）。

---

## 0. 定位转变（一句话锚点）

**From:** 一个押注「企业数据长期分散 + 在位平台不及时跟进」的、可防御的商业法务数据平台。
**To:** 一个**现在就能跑、聚焦、可贡献**的开源「AI 数据治理层」，目标是**积累工程经验 + 行业名望 + 服务真实小众需求**。

**北极星场景（产品的组织原则）：**

> **「小问题别来问我，去问你的 AI——它只会告诉你，你*该看的那一部分*。」**

把一个被反复打扰的领域专家（法务/HR/IT/财务），从「人肉查询 API」变成「自助式、按权限安全限定的 AI 端点」。

**为什么这个转向是解药而非退路：** 商业目标下的两个致命风险——①飞书会把治理延伸到 AI 边界，②今天 TAM 太薄——**在开源/名望目标下几乎不相关**。开源项目不需要护城河；它需要的是「聚焦 + 好用 + 时机对」。飞书两年后做了这功能？无所谓，那时你已经发布、学到、立了名，且中间服务了一批小团队。**你不再赌市场，只赌「把一件被低估的事做漂亮」。**

---

## 1. 现状分析

**代码现状：** v1.4 其实**已经造好了内核**——最小披露（v1.3）、服务端 Agent 授权（v1.4 `agent_query`）、审计（`audit_events`/`audit_disclosures`）。但 v1.5/v1.6 正在往**平台方向膨胀**：admin 录入 UI、导入落库管线、主体 MDM、Schema 迁移框架。

**关键诊断：v1.5/v1.6 在为「错误的目标」优化。** 它们优化的是「成为一个可防御的数据平台」——而这个目标本身已被否决。在新目标下，这部分工作从「资产」变成「负债」（维护成本、注意力黑洞、违背开源项目必须聚焦的铁律）。1.6.0 那个「没有迁移框架就用不了真实库」的阻塞，是平台路线**自己生出来的债**——而新方向根本不需要持有数据，这笔债可以直接勾销。

**两层结构复盘：**
- **事实层**（~6 表：projects/contracts/licenses/risks/entities/aliases）= 数据本身，**应交给数据源**（飞书/表格/库），不该自己持有。
- **治理层**（~12 表：users/groups/grants/access/keys/audit×3/agent×3）= **唯一的内核**，是北极星场景里「只告诉你该看的那部分」得以成立的全部理由。

北极星场景 = 治理层（「安全限定」）+ `agent_query` 自然语言路径（「去问 AI」）。**这两根柱子留，其余全砍。**

## 2. 第一性原理

**不可约的问题（三段，缺一不可）：**
1. 一个领域专家是重复性问题的**串行瓶颈**——别人不停来问「这项目到期没」「这合同对方是谁」「我能不能看 X」。
2. 他**不能直接把原始数据甩出去**自助，因为不同的人有权看不同的子集。
3. AI 本可以代答这些问题，但**把 AI 朴素地接到数据上，会让权限边界当场塌掉**（AI 走一个服务账号，看到一切）。

**因此，不可约的产物只有一个：** 一个**坐在 AI 与数据之间**的东西，它 (a) 知道**是谁**在问，(b) 在**字段级**强制此人可见的范围，(c) 让 AI 在这个范围内自由作答，(d) **记录披露了什么给谁**。

**仅此而已。** 存储、录入、归一、UI、同步——全部非本质，且全部已被别的工具做得更好（飞书多维表格/ETL/MDM/低代码平台）。**第一性原理把产品收缩成一个「策略 + 审计」的瘦网关，外加一个自然语言问答入口。**

## 3. SWOT（按「开源/现在」目标校准，不是按商业目标）

### Strengths（优势）
- 内核**已经造好**（v1.4）：工作是「删到内核」，比「从零造」快得多。
- 创始人 = 真实用户（in-house legal），自带 dogfooding 与可信度。
- **MCP 权限/授权在生态里是公认的空洞**，此刻发布一个像样的实现，时机上 reputationally 极佳。
- 北极星场景**普世易懂**，一个 GIF 就能讲清，利于传播。

### Weaknesses（劣势）
- 现有代码背着大量平台包袱（admin UI/ingest/MDM/迁移），**重构 = 删自己写的代码**，有沉没成本的情绪阻力。
- 个人/小团队产能，**一旦发散就会死**。
- 「通用框架 vs 法务专用」未决：太通用 → 没人的痛；太专 → 贡献者池小。
- 还没有社区与分发渠道。

### Opportunities（机会）
- MCP 生态正在爆发，auth/permission 是已知缺口——**顺着 MCP 浪潮填一个真实的洞**。
- 「自托管、数据不出内网的 AI 访问」对隐私/合规人群（中国 + 欧盟 + 强监管行业）有强共鸣，这是**价值观对齐、乐于采用开源**的人群。
- **连接器接口 = 社区贡献引擎**：出接口 + 2 个样例，别人帮你接飞书/钉钉/Postgres/Notion。
- 有机会成为 MCP-auth 讨论中被引用的**参考实现**。

### Threats（威胁）
- 平台吞并（飞书/MCP 原生 auth）——**但在新目标下影响降到「低」**，请主动给它重新评级。
- MCP 协议自身标准化 auth：可能让做法过时，**也可能是机会**（对齐/贡献标准）。
- 注意力稀缺：MCP 开源项目很多，出头靠**锋利的故事 + 好文档**。
- **范围回潮**：把你拖回造平台的那股引力（就是它生出了 1.5/1.6），是最现实的内部威胁。

---

## 4. 存在价值

- **对用户：** 把自己从「人肉查询 API」变成「自助、按权限安全限定的 AI 端点」。**夺回时间，同时不泄露不该泄露的字段。** 这正是原始设计文档里那个真实痛点（创始人自己最痛），现在被正面解决。
- **对生态：** 提供 MCP 缺失的那一层——**终端用户感知的字段级授权 + 披露审计**的干净参考实现。这是当下被讨论、未被做好的事。
- **对你自己（真目标）：** 一个聚焦、scoped 良好、时机对的开源作品；沉淀**可迁移的硬技能**（MCP、授权设计、审计/合规模式、连接器架构）；一个能拿出手的 portfolio 与名望资产——**且不依赖赢得任何市场**。

## 5. 改进方案（开发方向变更）

### 砍（Kill）— 造轮子或无用功
- **本地模式**：删。
- **admin 录入 / CRUD UI**：在和飞书抢录入，删。
- **导入落库管线（ingest-to-own）**：改为 read-through，不持有数据 → **1.6.0 的迁移框架阻塞随之消失**。
- **主体 MDM / 合并**：删。
- **冻结整个 1.6.0 平台路线。**

### 留并磨亮（Keep）— 两根柱子
1. **`agent_query` 自然语言路径**（「去问 AI」）—— 把 NL 问答做到好用，这是场景的左半边。
2. **按用户的字段级披露治理 + 审计**（「只看该看的」）—— 场景的右半边，差异化皇冠。

### 改架构（Change）— 从「事实源」到「瘦治理网关」
- 数据**留在源**（CSV/SQLite 零摩擦演示用；飞书多维表格做真实在岸案例）。服务端只持有 **策略 + 审计 + 一个薄 catalog**。
- **数据源做成可插拔连接器接口**，但只**官方维护 1–2 个**（CSV/SQLite + 飞书）。其余交给社区——**连接器接口就是贡献面**。
- **策略模型是皇冠上的钻石**：一份**声明式、可进 Git、可测试**的权限/披露规格（role → domain → field → allow/deny）。这是别人会 star、会复用的核心 artifact，**文档和测试要做到示范级**。

### 分阶段（务实顺序）
1. **重定位**：重写 README（按北极星场景）；删本地模式；冻结 1.6.0。
2. **抽内核**：服务端改 read-through over 可插拔源；出 CSV/SQLite 参考连接器；策略落成声明式文件。
3. **磨柱子**：终端用户身份穿透 + 字段级披露强制 + 审计，配示范级文档。
4. **真实证明**：接一个飞书多维表格连接器，跑通「不出内网」的真实案例。
5. **分发**：录一个北极星场景的 GIF/demo；写 examples + 贡献指南（连接器接口 + 策略 cookbook）。

## 6. 重要、但你没问的事

1. **成功指标必须换一套。** 商业目标用「营收/可防御性」——正是它逼出了 1.5/1.6 的过度工程。新目标的指标是：**GitHub star、外部贡献的连接器数、被 MCP-auth 讨论引用、N 个团队自托管、以及你自己每天在用**。把旧指标从脑子里删干净，否则引力会把你拖回造平台。

2. **最难的一步是删自己写的代码。** 沉没成本会以「可惜了」「再留留」的形式偷袭你。**每一行不服务北极星场景的代码都是负债。** 立一条硬规矩：砍之前先问「这行能追溯到那句北极星吗？」

3. **范围回潮是头号内部威胁。** 把「不存数据、不做录入、不做归一」写成项目的**明文非目标**，钉在 README 和 CONTRIBUTING 里，挡住自己也挡住贡献者的「顺手加个功能」。

4. **主动拥抱 MCP auth 规范，别假装它不存在。** 协议层正在演进 authz——**对齐它、甚至给它贡献**，比和它对着干安全。如果实现成了规范讨论的参考，那是名望的最大化路径。

5. **唯一需要现在拍板的决策：「通用框架 vs 法务专用」。** 建议：**通用的治理内核（permission-aware MCP server）做框架，法务做旗舰示例（reference domain）**。这样最大化复用、star 与贡献者池，同时保留最懂的领域作为可信门面。这也意味着项目可能要从 `legal-mcp` 改名到一个更中性的名字，`legal` 作为 `examples/` 里的样板。

6. **承重假设也换了一条，且温和得多。** 商业版的承重假设是「企业永远分散 + 平台永远慢」（在赌反共识）。**开源版的承重假设只是：「在 MCP 把按用户的字段级授权 + 审计变成标配之前，存在一段窗口期，里面有一批重视数据不出内网的小团队愿意自托管一个能用的方案。」** 这个假设温和、当下成立、且你只要在窗口期内交付就赢——不需要它永远成立。

---

**一句话收尾：** 你之前在为「赢得一个市场」做工程，所以越做越重；现在改为「把一件被低估的事做漂亮、并因此成长和立名」——同一个内核，**第一次有了和它复杂度匹配的目标**。砍掉两根柱子之外的一切，产品就从「自我怀疑的平台」变成「定义清晰的治理层」。
