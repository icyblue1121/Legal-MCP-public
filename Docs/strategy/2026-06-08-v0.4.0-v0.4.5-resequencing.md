# v0.4.0 / v0.4.5 re-sequencing — sequential development plan

> Re-orchestration decision (2026-06-08, rev). **This supersedes the earlier
> "v0.4 identity ships first, v0.4.5 arbitrary-domain" sequencing.** The order is
> now inverted, and the one cross-dependency (`by_owner` row scope) is moved to
> live in the same version as the identity it consumes. Net effect: two
> self-contained versions developed strictly in sequence, neither shipping a
> feature whose dependency lands in a later version.
>
> North star unchanged: *小问题别来问我，问 AI；AI 只回答你有权知道的那一部分。*

## Why the order flipped

| Fact (verified against code 2026-06-08) | Consequence |
|---|---|
| Arbitrary-domain work (old v0.4.5 §A non-owner, §B, §C, §D) does **not** depend on identity passthrough — per-user grant resolution already works via `identity.py` api-key→user. | It is fully unblocked **today**. |
| That work has a complete design + acceptance criteria (`2026-06-08-arbitrary-domain-authorization-plan.md`). | It is execution-ready. |
| Identity passthrough (old v0.4) has only a 4-line sketch (gateway plan §522-529); no detailed design. | It needs a design pass before execution. |
| Only `record_scope: by_owner` ("only see your own rows") consumes the passed-through identity. | It is the single coupling point — move it to the identity version. |
| User's recurring priority: "连接本地/其他位置的数据源 + 后台可见". | Data-source capability should ship first. |

**Decision:** data-source capability becomes **v0.4.0** and ships first; identity
passthrough + `by_owner` becomes **v0.4.5** and ships second.

## Version split

```
v0.4.0  任意数据源：接入 · 授权 · 后台可见        ← ships first, no identity dependency
        = old v0.4.5 minus by_owner
        A(record_scope: none|by_governed_code) + B(catalog-driven metadata)
        + C(admin data-source & permission UX) + D(scaffold-connector)

v0.4.5  身份穿透 + by_owner 行级范围              ← ships second
        = old v0.4 identity work + the by_owner row scope that consumes it
        identity interface + HTTP-header/reverse-proxy + (OIDC optional)
        + record_scope: by_owner
```

The detailed §A/§B/§C/§D design lives in
[`2026-06-08-arbitrary-domain-authorization-plan.md`](2026-06-08-arbitrary-domain-authorization-plan.md)
— this doc is the **sequencing + phase plan**; it does not re-derive that design,
it only re-labels it v0.4.0 and orders the phases.

## Policy-posture decision (2026-06-07, rev2): remove the git YAML gate entirely

The product's users are non-technical; the admin console manages projects, data
sources, and domain/field permissions. The git YAML policy was a *second* field
gate (a version-reviewed disclosure ceiling) ANDed on top of the DB grants.

**Decision (supersedes rev1's "keep as optional/advanced"): remove the git YAML
policy from the product entirely.** The console (DB `permission_grants`) becomes
the *single* authorization path — there is no parallel, file-based gate.

Rationale: DB grants are already **default-deny by construction** (no grant → no
disclosure; `authorize_fields`), so the gateway is fail-closed *without* any YAML.
A dormant, optional, file-based second path that most operators never use is not
free: it is **attack surface and, worse, cognitive surface** — a non-technical
operator cannot reason about risk in a policy file they do not understand, and an
invisible parallel gate can mask a misconfiguration or create a false picture of
"what the system actually does". Legibility wins: *what you see in the console is
the whole authorization truth.*

Tradeoff accepted, with eyes open: removing the second gate makes **DB-grant
correctness the entire security boundary** — there is no independent backstop
against DB tampering or a grant bug. Compensation is folded into the removal work
below (the leakage red-team test becomes the sole guarantor; DB integrity + audit
+ backups are the trust root). The "change-controlled, git-reviewable ceiling"
selling point is given up deliberately; if a genuine two-trust-level customer ever
appears, the answer is a **console-managed global ceiling (owner-role-gated)**,
not a return to git files.

**Removal work (its own item; sequence BEFORE deleting, then delete):**
1. *Prove console self-sufficiency first.* Reproduce every legal-demo disclosure
   outcome with DB grants alone (policy absent) — domain + field + group/user +
   row scope. This is the precondition that makes removal safe.
2. *Then delete the gate and its surface:* the `policy` parameter + `_policy_*`
   helpers in [query_authorization.py](../../src/legal_mcp/query_authorization.py);
   `policy=` threading in
   [connector_retrieval.py](../../src/legal_mcp/connector_retrieval.py) /
   `search_tools.execute_search_plan` / `agent_graph`; the `Policy` /
   `policy_config` module; the CLI `--policy` flag and any doctor checks;
   `examples/policies/`; and all policy-specific tests.
3. *Strengthen the remaining guarantor:* extend the disclosure-leakage red-team
   test to be the sole gate's full coverage (it already exercises the DB path).
4. *Rewrite the security narrative:* `Docs/strategy/threat-model.md` and
   `commitment-matrix.md` drop "defense in depth / two gates" and state a single,
   legible, console-managed gate with DB integrity as the trust root.

Sequencing: this rides **with Phase 2 (§C)** — step 1 is exactly the §C "console
is self-sufficient" acceptance; steps 2–4 land once §C proves it. No code this
turn.

## v0.4.0 — 任意数据源：接入 · 授权 · 后台可见

Goal: any connected source (SQLite / Feishu / future) can be authorized at
**domain and field level** (and row level via `by_governed_code` / `none`),
driven by the table's real content, and managed entirely from the admin console
— no code edit to add a domain. No identity passthrough required.

### Phase 1 — Generalized record scope + catalog-driven metadata (§A + §B)

The unblocker. Removes the two hard-coded project assumptions that block
non-project tables.

- **A. record_scope on the connector domain.** A domain declares
  `record_scope: {mode: none | by_governed_code, field}`. The gateway resolves
  scope per domain from this declaration instead of the constant
  `_DEFAULT_SCOPE_FIELD = "project_code"` ([connector_retrieval.py:46,56,116-132](../../src/legal_mcp/connector_retrieval.py)).
  - `mode: none` → no row post-filter; domain/field grant is the only gate.
  - `mode: by_governed_code` → exactly today's `project_code` behavior, now declared.
  - `mode: by_owner` → **reserved, declared as invalid/unsupported in v0.4.0**
    (loud error if configured), implemented in v0.4.5.
- **B. Per-domain metadata from the catalog, not code.** Derive identity fields
  from `ConnectorField.is_identity` (already declared, [connectors/base.py:37](../../src/legal_mcp/connectors/base.py));
  make relationship-filter and `cross_domain` participation per-domain catalog
  attributes (empty by default), replacing the hard-coded
  `PROJECT_IDENTITY_FIELDS` / `LICENSE_IDENTITY_FIELDS` /
  `PROJECT_RELATIONSHIP_FILTER_FIELDS` / the `{contract,license,risk}` set / the
  cross_domain participant list ([query_authorization.py:14-17,57-83,135-144,259-266](../../src/legal_mcp/query_authorization.py)).
  Legacy 3 domains keep current behavior by declaring the same metadata.

**Verify (gate to Phase 2):**
- A1 — a `mode: none` non-project table fixture queries end-to-end with
  field/domain authorization enforced and no project assumption in the path.
- A2 — `mode: by_governed_code` reproduces today's behavior; the existing
  mixed-source crown-jewel test passes **unchanged**.
- B1 — a brand-new domain (declare fields only, **zero** `query_authorization.py`
  edit) authorizes filter/return fields correctly; test asserts no code edit.

### Phase 2 — Admin: data sources, assignable permissions, per-user grant (§C)

**Scope decision (2026-06-07): consolidate the legacy `/admin/database` page into
this Data Sources view (option "改造合并").** The current `/admin/database`
([admin_database.py](../../src/legal_mcp/admin_database.py)) is a pre-pivot
artifact: it browses raw rows of the bundled SQLite demo (project codes, names,
stages, legal_bp…), aggregates entities, and offers a spreadsheet **import** box —
all of which contradict the post-pivot non-goals (no data-entry/CRUD, no BI, the
gateway does not own/browse business data). It is redefined here.

**The page's new purpose** — let an admin understand and operate the gateway's
data plumbing, *without ever showing a field's actual values*:
1. which data sources are currently connected (type: SQLite / Feishu / future);
2. what data each serves — its domains, and per domain the **field names**
   (identity/sensitive marked) and `record_scope` mode — **names and metadata
   only, never row values**;
3. which permissions are configured, and to which **user groups / users**, per
   domain/field;
4. **add a new connection** or **disconnect** an existing source.

**What is removed from the page:** the per-row project/contract/license content
table, the aggregated-entities table, and the xlsx/csv import box. (The demo-data
browse, if still wanted for the example, moves out of the operator console.)

- **C0.** Inject `ConnectorSetup` into `LegalMCPAdminServer` at construction
  (admin server does not hold it today — verified).
- **C1.** **Data Sources** view (replaces the database page body): every
  connected source, its type, the domains it serves, each domain's field *names*
  (identity/sensitive marked) and `record_scope` mode. No field values rendered.
- **C2.** **Assignable-permissions** catalog drives the permissions form,
  replacing the hard-coded domain `<option>` dropdown
  ([admin_manage.py:470-497](../../src/legal_mcp/admin_manage.py)). Granting a
  non-existent domain/field becomes impossible.
- **C3 + C4.** Add nullable `user_id` to `permission_grants` (make `group_id`
  nullable; exactly-one-of check; [schema.sql:145-154](../../src/legal_mcp/schema.sql)).
  Grant a domain (optionally a single field, plus row-scope selector) to a
  specific **user**. Resolve effective grants as `direct ∪ groups'`. Add a
  per-user effective-permissions view. The Data Sources view surfaces, per domain,
  which groups/users currently hold a grant (purpose #3).
- **C5. Add / disconnect a source from the console** (purpose #4). **Design
  tension to resolve, not gloss:** the connector config is a *reviewed,
  git-committable YAML artifact* — the security posture is "only declared fields
  are queryable, config is a reviewed artifact", and runtime mutation of which
  fields a source exposes would undercut it. Reconciliation: the console manages
  source **enablement and credentials**, not field declarations.
  - *Disconnect* = disable an already-declared source (stop routing its domains;
    fail-closed: its domains simply leave the live catalog). Reversible.
  - *Add* = register a source whose **field declaration still comes from a
    reviewed draft** — produced by the §D `scaffold-connector` flow (Phase 3),
    reviewed, then enabled. The console never invents queryable fields on its own.
  - Persist en/disable state where the deployment can audit it; never write
    secrets into the page or the DB (credentials stay in env, as today).
  - This subitem **depends on Phase 3 (§D)** for the add path's draft generation,
    so C5's *add* may land after D; *disconnect* can land in Phase 2.

**Verify (gate to Phase 3):** C1 (view lists sources+domains+scope, no values),
C2 (form options come from live catalog), C3 (per-user grant allows that user,
denies a peer — integration test), C4 (effective-permissions view shows
direct ∪ group), C5-disconnect (disabling a source removes its domains from the
live catalog; a query to a disconnected domain is denied). Security check: the
consolidated page renders **no** business field value anywhere (assert in a test).

### Phase 3 — Content-driven scaffolding (§D)

- **D.** Add optional `describe_schema()` to `connectors/base.py` (SQLite via
  `PRAGMA`; Feishu via list-tables/list-fields). `legal-mcp scaffold-connector`
  emits a **draft** YAML from real columns (all fields, guessed identity, blank
  aliases, `record_scope.mode: none`). Human reviews → commits. Auto-gen never
  widens disclosure on its own.

**Verify:** D1 — `scaffold-connector` against any connector emits a valid draft
that, after a no-op review, loads and serves the table.

### v0.4.0 security gate (before tagging)

Extend `tests/test_disclosure_leakage.py` to cover an arbitrary (non-project)
domain: an unauthorized field's value never enters model context for the new
domain either. Must pass before v0.4.0 is tagged.

## v0.4.5 — 身份穿透 + by_owner 行级范围

Goal: the gateway authorizes by the **real person** asking, across realistic team
deployments — and `by_owner` ("you only see your own rows") becomes possible
because the row filter now has a passed-through identity to bind to. This is the
project's #1 technical risk (gateway plan §8.6), so it gets its own version and a
real design pass — not a rushed rider on the data-source work.

**Definition checkpoint (2026-06-09 review):** in v0.4.5, "身份穿透" means
the gateway resolves the end user's identity from a trusted upstream source and
uses that identity inside the gateway authorization model (`AccessContext` +
record scope). It does **not** mean delegating final-user authorization to Feishu,
SQLite, or any other backing data source. Source-side per-user delegation is a
larger v0.5+ design because it changes connector credentials, source permission
semantics, audit ownership, and failure modes. Concretely: the connector keeps a
**single shared service credential per source** — the gateway authenticates to
Feishu/SQLite as *itself*, never as the end user. That is exactly what keeps the
actor in the gateway's own audit trail (the end user is recorded by the gateway,
not by Feishu); source-side delegation is what would move audit ownership out of
our hands. This boundary is the line between a v0.4.5 increment and a v0.5+ rewrite.

**Preflight before Phase 1:**
- Resolve the current fail-open before adding another identity path. The contradiction
  is verified in code and is **live**, not nominal: `authorize_fields`
  ([policy.py:130](../../src/legal_mcp/policy.py)) returns *all* requested fields for
  a `legacy_shared_token` context, reachable in the live HTTP path via
  `AccessContext.legacy()` ([http_server.py:147](../../src/legal_mcp/http_server.py)).
  The docs even contradict *themselves* — `threat-model.md` and `identity-model.md`
  call it both "不豁免字段门 / fail-closed" *and* "全访问后门 (full-access backdoor)" —
  so the fix lands in both docs, not one.
  - **It is one branch across two gates.** The same `context is None or
    context.legacy_shared_token` test is in *both* `authorize_fields` (field gate)
    and `visible_project_ids` (row gate, [policy.py:58](../../src/legal_mcp/policy.py)).
    The legacy token bypasses field **and** row scope; the posture decision must
    cover both.
  - **Fix structurally, not per-flag.** Today "see everything" is a *fall-through*:
    `context=None` and `legacy_shared_token` land in the same allow-all branch as a
    default. v0.4.5's whole job is to add a *new* way to mint an `AccessContext`
    (the resolver seam) — so any new caller that produces a None/legacy-shaped
    context, or that forgets the HTTP-layer 401 guard currently saving `context=None`
    ([http_server.py:90](../../src/legal_mcp/http_server.py)), silently becomes full
    disclosure. Make `authorize_fields`/`visible_project_ids` **fail-closed by
    default** and make "unrestricted" an *explicit named capability* (admin role, or
    an explicit opt-in migration flag), never a None fall-through. Preferred posture:
    legacy token does not bypass ordinary field grants; full access is available only
    behind a documented, default-off opt-in flag for the migration window.
  - **✅ Implemented (preflight, this version).** `AccessContext` gained an explicit
    `unrestricted` capability; `legacy()` is fail-closed by default and only
    `legacy(unrestricted=True)` (gated by the new default-off `--legacy-token-full-access`
    CLI flag) restores full access; the stdio server now passes an explicit
    `AccessContext.local_operator()` instead of relying on the `None` default. Both
    security docs were de-contradicted. **Split decision:** the *live* hole (legacy
    token) is closed here; the `context=None` fall-through is **deferred to Phase 1**,
    because `None` is still the load-bearing stdio/local-library default and the HTTP
    path already 401s a missing context — flipping `None` to fail-closed belongs with
    the resolver seam (Phase 1), where every context becomes explicit and the
    accompanying test churn (callers passing an explicit `local_operator()`) is
    thematically coherent. New coverage: `test_policy.py` (legacy fail-closed default,
    opt-in full access, explicit local operator), `test_cli.py` (flag default off /
    opt-in), `test_http_server.py` (flag plumbing default off).
- Decide the canonical owner subject for row scope. The default should be
  `external_subject` when present, falling back to `email` only for local API-key
  pilots. `user_id` should be supported only for local/demo sources because it is
  not meaningful in external SaaS tables. *No schema migration needed:*
  `users.external_subject` (column + index) already exists
  ([schema.sql:101,292](../../src/legal_mcp/schema.sql)).

### Phase 1 — Identity interface design + resolver seam

Promote the 4-line sketch into a design. Define a pluggable `IdentityResolver`
seam with the upstream sources and their trust assumptions, and update
`Docs/strategy/identity-model.md`:
1. per-user API key (already works via `identity.py`) — baseline,
2. HTTP header injected by a trusted reverse proxy,
3. OIDC/OAuth minimal — **explicitly non-blocking for v0.4.5; expected v0.5 unless
   the header path leaves it nearly free.**

Implementation shape:
- Add a focused resolver module (for example `identity_resolver.py`) instead of
  growing `http_server.py`. The resolver returns a single `AccessContext` shape
  regardless of source.
- Extend `AccessContext` with the row-scope owner subject needed by `by_owner`
  (`external_subject` and/or an explicit `owner_subject`), while keeping the
  existing api-key path behavior-identical.
- Keep resolver precedence unambiguous: one request should authenticate through
  exactly one accepted identity source. If a request presents conflicting identity
  sources, reject it rather than choosing silently.
- **Harden the deferred `context=None` fall-through here.** Now that every entry
  point mints an explicit `AccessContext` through the seam (network paths) or an
  explicit `local_operator()` (stdio), flip `authorize_fields`/`visible_project_ids`/
  `can_query_content` so `None` is **fail-closed** rather than unrestricted. This is
  the second half of the preflight's structural fix; it is cheap to land here and
  absorbs its test churn (callers that relied on `None`=full-access pass an explicit
  `local_operator()`) alongside the resolver work.

**Verify:** design doc reviewed; resolver interface stable; existing api-key path
re-expressed through the seam with no behavior change; tests cover api-key success,
revoked/disabled user denial, and conflicting identity sources; `None` context is
fail-closed at all three gates.

### Phase 2 — HTTP header identity + reverse-proxy trust

Gateway accepts a configurable identity header **only** from a configured trusted
proxy (untrusted header is ignored/rejected). Document the trust boundary.

Concrete requirements:
- Configurable header name, for example `X-Legal-MCP-User` or deployment-specific
  equivalent. The value maps to `users.external_subject` first; email fallback is
  allowed only when explicitly enabled.
- Configurable trusted proxy source, checked against the HTTP peer address. A
  direct client-supplied identity header from an untrusted source must be rejected
  fail-closed, not silently trusted.
- Header identity requires the upstream proxy to have already authenticated the
  human. Legal-MCP trusts the proxy boundary, not the raw header.
- Audits must record which identity source resolved the request (api-key vs trusted
  header) without storing bearer tokens or raw secrets.
- The conflict check must treat the **legacy shared token as one of the identity
  sources** — it *is* a bearer token. "Two identity sources present → reject" must
  fire for `legacy-token + trusted header`, not only `api-key + header`.

**Verify:** header from trusted source resolves to the intended active user;
unknown/disabled users fail closed; spoofed header from an untrusted source is
rejected; a request carrying both a bearer API key and a conflicting trusted
identity header is rejected; a request carrying the legacy shared token **and** a
trusted header is likewise rejected.

### Phase 3 — OIDC minimal (defer unless already paid for)

**Decision (v0.4.5, 2026-06-09): DEFERRED to v0.5.** OIDC was *not* "nearly free."
The core package is deliberately **zero-runtime-dependency** (`pyproject.toml`
`dependencies = []`) — a self-hosted security gateway with no supply-chain surface
in its core. A real OIDC path needs either the *first-ever* runtime dependency
(`PyJWT[crypto]` for JWKS/RS256 validation), hand-rolled RSA verification
(security-critical, declined), or a full browser redirect flow (callback routes +
session cookies + PKCE — a new HTTP surface). None of those reuse the Phase 2 header
path for free, so per this section's own rule ("otherwise defer to v0.5") OIDC moves
to v0.5. The resolver seam already accepts a future `JwtBearerSource`/OIDC source
with no change to `by_owner` or the trusted-header trust model, so the deferral
costs nothing structurally. The v0.4.5 tag gate does not depend on it.

OIDC/OAuth is not part of the v0.4.5 tag gate. It can land in v0.4.5 only if it
reuses the same resolver seam without changing the `by_owner` implementation or
trusted-header trust model. Otherwise defer to v0.5.

**Verify if included:** OIDC subject maps to `users.external_subject`; unknown or
disabled users fail closed; the callback/session path does not weaken API-key or
trusted-header behavior.

### Phase 4 — `record_scope: by_owner` (closes the v0.4.0 reserved item)

**✅ Implemented (v0.4.5).** `RecordScope` gained `subject` (the gateway identity
attribute, one of `external_subject` / `email` / `user_id`); `by_owner` parses at
config load (explicit `field` required, `subject` defaults to `external_subject`).
The own-scope resolver `policy.record_owner_subject` returns the subject string or
`None`-means-**zero-rows** (NOT the `visible_project_ids` `None=all` sentinel) — so
legacy / anonymous / unrestricted-but-unmapped get no rows. The owner equality is
pushed **down** into the connector filters (overriding any client filter on the
owner field), fixing the false-empty; a defense-in-depth post-filter drops any
non-owner row a broad connector returns; the owner field is fetched for that check
but stripped from the projection unless explicitly requested. The leakage red-team
test covers a `by_owner` domain (a peer's row value never enters context).
`by_governed_code`'s set-membership stays a documented post-filter fallback (its
limit-before-filter false-empty is noted, since it can't be pushed through the
equality-only connector).

Now that identity is passed through, implement the `by_owner` mode reserved in
v0.4.0 §A: row field bound to the requesting user's identity. A grant for a
`by_owner` domain means "your own rows".

Implementation requirements:
- Extend `record_scope` with the comparison subject, for example
  `record_scope: {mode: by_owner, field: owner_email, subject: email}` or
  `subject: external_subject`. The source row field (`field`) and gateway identity
  attribute (`subject`) must both be explicit in config, except for a documented
  safe default.
- `by_owner` requires an identified user and a non-empty configured owner subject.
  Anonymous, legacy, or unmapped users get zero rows.
- **Fail-closed red line — do NOT reuse the `by_governed_code` scope path.** The
  existing sentinel `visible_project_ids` / `record_scope_project_ids` returns
  `None = unrestricted` for legacy/None contexts
  ([policy.py:58](../../src/legal_mcp/policy.py),
  [search_tools.py:262](../../src/legal_mcp/search_tools.py)). If `by_owner` naively
  routes through it, a legacy/anonymous context → `None` → unrestricted → **every
  owner's rows leak**. `by_owner` needs its *own* scope resolver whose "no subject"
  result is the **empty set** — the opposite default from `None=all`. This is the
  natural-but-wrong mistake to guard against in review and tests.
- **Push the owner predicate down — it is cheap and needs no new connector surface.**
  `by_owner` scope is a single equality (`owner_field == subject`), which already
  fits the equality-filter pushdown (`_equality_filters` / `ConnectorQuery.filters`,
  [connector_retrieval.py:159](../../src/legal_mcp/connector_retrieval.py)): inject
  `{owner_field: subject}` into `filters` before `connector.query`, and the source
  applies filter-then-limit correctly. (This is *easier* than `by_governed_code`,
  whose set-membership `in (...)` scope can't be expressed as an equality and is why
  that path post-filters.)
- **Pre-existing bug to fix, not just avoid:** the shipped `by_governed_code` path
  already has the "limit before filter → false-empty" failure mode codex warned
  about. [connector_retrieval.py:110-128](../../src/legal_mcp/connector_retrieval.py)
  calls the connector with `limit=plan.limit` + the plan's filters, applies scope as
  a Python *post-filter* (line 121-122), then re-truncates (line 128) — so a user
  whose rows sit past the source's first `limit` rows gets a false-empty *today*.
  v0.4.5 fixes the pattern (pushdown for the equality cases), not only the new
  `by_owner` mode. Post-filter survives only as a documented fallback for connectors
  that genuinely cannot push down, with a test covering the failure mode.
- Fetch the owner scope field only for authorization. Do not project it into the
  model response unless it was explicitly requested and field-authorized (the
  projection-stripping at [connector_retrieval.py:126](../../src/legal_mcp/connector_retrieval.py)
  already does this; reuse it).
- **Audit the resolved subject, not just the source.** Beyond Phase 2's "which
  identity source," record the *subject value* used to scope the rows, so a later
  reviewer can verify "user X saw only X's rows."

**Verify:** a `by_owner` table returns only the requester's rows; a peer sees only
theirs; unmapped/legacy contexts see no rows (assert the legacy/None path does
**not** fall through to `None=all`); the connector query receives the owner scope
predicate **before** pagination (assert the user's row past the first `limit`
source rows is still returned, i.e. the false-empty is fixed); the disclosure-leakage
test covers a `by_owner` domain and proves another user's row values never enter
model context.

## Sequencing summary

```
v0.4.0  Phase 1 (A+B)  → verify A1/A2/B1
        Phase 2 (C)    → verify C1/C2/C3/C4
        Phase 3 (D)    → verify D1
        security gate  → tag v0.4.0
                                │  (data sources fully usable, no identity needed)
v0.4.5  Phase 1 identity design+seam
        Phase 2 HTTP header + reverse proxy
        Phase 3 OIDC minimal (deferred unless nearly free)
        Phase 4 by_owner row scope  → tag v0.4.5
```

Strictly sequential. v0.4.0 is self-contained and shippable on its own; v0.4.5
adds the identity dimension and only then unlocks `by_owner`.
