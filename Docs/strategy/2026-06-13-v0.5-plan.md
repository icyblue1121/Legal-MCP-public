# Legal-MCP v0.5 路线图：自助数据源接入 + planner 检索增强

> 配套：`2026-06-06-permission-aware-mcp-gateway-plan.md`（北极星）、`identity-model.md`、
> `threat-model.md`、`2026-06-08-arbitrary-domain-authorization-plan.md`。
> 立项日期：2026-06-13。前置版本：`v0.4.9`（505 绿，已 tag）。

## 0. 一句话目标

**让非技术的法务/运维用户，在管理后台「一键」把自己的在线或本地表格接进网关并开始问答——
而授权、行级/字段级裁剪、审计这套既有保证一行不松。** v0.5 把「接入数据源」从
「改 YAML + 重启」降为「后台点几下 + 人工 review 字段」。

## 1. 与原计划的差异（决策已拍板，2026-06-13）

原 TODOS 的 v0.5 = 「contributor surface（连接器指南/policy cookbook/安全文档）」，且 v0.4.5
把 OIDC 推迟到 v0.5。经与负责人确认，v0.5 重新定义为下面三柱，**OIDC 自研被移除**：

| # | 原设想 | v0.5 实际 | 决策依据 |
| --- | --- | --- | --- |
| 1 | 自研 OIDC/OAuth 登录 | **不自研**。文档化「前置 OIDC 反代 → 可信头 → 现有 `TrustedHeaderSource`」的企业 SSO 落地路径 | 需要强登录的用户用成熟反代（oauth2-proxy / Authelia / nginx-oidc）自行解决；现有 seam 已支撑，自研只会引入第一个运行时依赖与浏览器跳转复杂度 |
| 2 | 官方深做 1–2 个连接器 | **重心改为「控制台一键接入框架」**：后台自助连接在线/本地数据源，覆盖主流在线文档与本地表格 | 用户要的是「加按钮 + 引导自助接入」这套能力，而非某两个固定源 |
| 3 | （无） | **planner 检索增强**：修 `eq` 假空、适应更宽泛字段与问法 | dogfood 反复出现的真实痛点（`MOON的法务BP是谁` 假空） |

**数据范围（决策）**：在线 + 本地的**结构化 / 半结构化表格**——CSV / XLSX / JSON /
Markdown frontmatter。PDF / Word 正文检索**不在 v0.5**，连同「权限模型能否治理 RAG 向量
检索结果」这一研究问题，一起列为 **v0.6+ 明确探索项**（见 §7）。

## 2. 非目标 / 边界（在既有非目标上叠加）

- **仍不自研 OIDC/OAuth**——交给前置反代/IdP；网关只消费可信头。
- **仍不做非结构化正文检索 / 文档知识库 / OCR**（v0.6+ 探索，且需先回答权限治理问题）。
- **「一键接入」≠ 任意源零代码**。每个**源类型**仍需一个连接器实现；框架让「配置 → 自省字段
  → 人工 review → 启用 → 授权」一键化，**新源类型由连接器作者扩展**，不是凭空支持所有 SaaS。
- **默认拒不变**：后台新接入的源，默认 `record_scope: none` + **零 grant** →
  在 admin 显式授权前**无人可查**。接入动作绝不自动放宽披露。
- **不拥有数据**：本地文件连接器是**读直通**（read-through）+ 自省，不是入库/MDM/导入平台。

## 3. 现状盘点：v0.5 三柱各自复用的 seam

> 由 codegraph 通读核对（2026-06-13）。具体行号会漂移，以符号名为准。

**身份（柱三的反面——确认无需改）**
- `identity_resolver.py`：`IdentitySource` 协议（`is_present` / `resolve`）、`resolve_access_context`
  单源择一 + `ConflictingIdentitySources` fail-closed；`BearerTokenSource`、`TrustedHeaderSource` 已就位。
- `http_server.py` `_identity_sources()`：按 CLI flag 组装身份源——**这正是未来 OIDC 源的接入点，
  v0.5 不动它，只在文档里说明如何用反代替代**。

**连接器（柱一/柱二的地基）**
- `connectors/base.py`：`DataConnector`（`catalog` / `query` / 可选 `describe_schema` / `fast_intents`）、
  `ConnectorDomain` / `ConnectorField` / `RecordScope` / `ConnectorQuery` / `ConnectorFilter`（含 `or_fields`）、
  `SourceTable`（自省结果）。
- 已有实现：`connectors/sqlite_demo.py`（含 `describe_schema` via PRAGMA）、
  `connectors/feishu_bitable.py`（在线多维表样板；算子翻译 eq/contains/in/or_fields）。
- `connector_config.py`：`build_connector_setup(data, ...)` 从 **YAML dict** 构建
  `CompositeConnector`；`ConnectorSetup.sources_for(domain)` 多源优先级；`connectors/composite.py` 路由。
- `scaffold.py`：`scaffold_connector_config(connector, ...)` 调 `describe_schema()` 产出草稿 YAML
  （全列、首列猜 identity、`record_scope: none`），供人工 review——**这是「一键接入」自省+草稿环节的现成内核**。
- `admin_database.py`：Data Sources 视图 + `data_source_state` 启停（schema v20）。
  **缺口：只能 disconnect/reconnect 已在 YAML 声明的源，不能在后台「新增」一个源。**

**planner（柱三）**
- `agent_graph.py` `plan_query` / `_planner_system_prompt` / `_classify_with_ai_planning`（重试上限 `MAX_PLAN_ATTEMPTS`）。
- `query_plan.py`：`QueryPlan`（含 `data_source`）、`QueryFilter`、`VIRTUAL_IDENTITY_FIELD = "identity"`、
  `SUPPORTED_OPERATORS`（含 `is_empty` / `date_*`，但**连接器侧未实现翻译**）。
- `identity_match.py`：`rank_identity_rows`（精确优先 / 候选排序），`connector_retrieval.py` 把
  `identity` 虚拟过滤展开成 `or_fields`。
- **已知未修 bug**：planner 偶发 emit `name eq <token>`（大小写敏感、单字段猜）而非 `identity contains`，
  导致授权成功却 `no_rows` 假空（v0.4.9 故意未做「源内零结果改写」）。

## 4. 柱一 —— 控制台一键接入框架（v0.5 核心）

把「源配置」从**静态 YAML、启动期加载**升级为**DB 持久化、运行时可增**，并在后台提供向导。

### A0. 运行时源注册表（架构基座）
- 新表 `data_sources`（schema v23）：`id, name, type, status(draft|active|disabled),
  config_json, secret_ref, created_by_user_id, created_at, updated_at`。
  `config_json` 存经人工 review 后的域/字段/identity/record_scope/别名声明（即 scaffold 草稿的落库形态）。
- `connector_config` 增加「从 DB 行构建连接器」的路径，与现有「从 YAML dict 构建」**共用**
  `build_connector_setup` 的源分派逻辑（`type` → 工厂）。最终 `CompositeConnector` 由
  **YAML 声明源 ∪ DB active 源** 合并而成。
- **热生效**：`_catalog_for_database`（agent_graph）已是按请求重建 catalog + 读 `data_source_state`，
  把「DB active 源」纳入同一重建路径即可让新源**免重启**生效；为性能加一层带失效的缓存（源表 `updated_at` 作 key）。
- *验收 A0*：后台插入一行 active 源 → 下一次 agent 查询的 catalog 立即含该源的域，无需重启。

### A1. 「添加数据源」向导（admin UI，server-rendered，沿用 v1.5 admin 设计系统）
四步，全程**只暴露字段名/元数据，永不显示行值**（与 Data Sources 视图同纪律）：
1. **选类型**：在线（Feishu / 腾讯文档智能表格 / 钉钉多维表 / Notion / Airtable）｜本地（CSV / XLSX / JSON / Markdown-frontmatter）。
   列出的类型 = 已注册连接器工厂；未实现的类型不出现（诚实，不画饼）。
2. **连接**：在线 → 填 app_token/资源 id + 凭据（见 A2）；本地 → 上传文件或填服务器路径。
3. **自省**：调 `describe_schema()` 列出表与列（在线靠各源 list-fields API，本地靠解析表头）。
4. **review + 启用**：勾选要暴露的字段、标 identity / sensitive、设 `record_scope`(none|by_governed_code|by_owner)、
   填别名 → 写入 `data_sources(status=active)`。**默认全字段不暴露敏感、record_scope=none、零 grant。**
- *验收 A1*：用一个本地 CSV 走完四步 → 该域出现在 catalog；在未授权前，任意非 admin 用户查它得授权拒绝（非空值）。

### A2. 凭据存储（**安全决策点**，需负责人拍板）
| 方案 | 一键程度 | 安全性 | 影响 |
| --- | --- | --- | --- |
| **(推荐) env 引用** | 中（仍需在部署侧设 env） | 高，秘钥不入库 | `secret_ref` 存 env 变量名；后台不接触明文秘钥 |
| 加密入库 | 高（后台直接贴 token） | 中，引入主密钥管理 | 需 `LEGAL_MCP_SECRET_KEY` 主密钥；威胁模型新增「DB 泄漏 = 秘钥泄漏」面 |
- 建议：**默认 env 引用**，把「加密入库」作为显式 opt-in（部署设了主密钥才开放后台贴 token），
  并在 `threat-model.md` 增补对应威胁行。

### A3. 安全护栏（**必须**，否则一键接入会变成一键越权）
- **仅 `admin` 可加/改/删源**；接入/变更/启停均为高价值审计事件（落 `audit_events`）。
- **新源默认 `record_scope: none` + 零 grant** → 默认拒；只有 review 时显式声明的字段可查（不自动放宽）。
- scaffold 草稿**永不**把列标成已授权；identity/sensitive 由人确认。
- `threat-model.md` 增补：**「疏忽/恶意 admin 接入一个含敏感列的源并误授权」** 的边界——
  靠「默认拒 + 字段需显式 grant + 接入即审计 + 泄漏红队测试覆盖任意域」共同兜底。
- 泄漏红队测试（`tests/test_disclosure_leakage.py`）扩到**一个经向导接入的 DB 源**，证明
  over-request / projection / denied-only 三类攻击对运行时新增源同样 fail-closed。

### A4. Data Sources 视图升级
- 现「连/断」基础上加：**+ 添加数据源** 入口、每个源的 draft/active/disabled 状态、删除（解绑）源。
- 删除源 = 从注册表移除 + 其域离开 catalog（`unsupported_domain`），与现 disconnect 行为一致。

## 5. 柱二 —— 连接器扩展（让框架「有料可接」）

框架（柱一）只在有连接器时才有意义。v0.5 配套交付：

### B1. 本地文件连接器 `local_file`（**新建**，覆盖 CSV / XLSX / JSON / MD-frontmatter）
- 实现 `DataConnector`：`describe_schema()` 解析表头/键得列；`catalog()` 由 review 后的声明生成；
  `query()` 执行 `ConnectorFilter`。
- **半结构化映射**：JSON 数组 / JSONL / YAML 列表 → 行；每条记录的键 → 列。Markdown frontmatter →
  每文件一行、frontmatter 键为列（正文**不**进入——正文是 v0.6 范围，这里明确切线）。
- **算子下推实现选择（决策）**：把加载的表落入**临时/内存 SQLite** 再复用 `sqlite_demo` 的算子翻译
  （eq/contains/in/or_fields + `like`/nocase），**优于**手写内存过滤——一次实现拿到全部算子且与现有路径一致。
- *验收 B1*：一个 XLSX 和一个含 frontmatter 的 `.md` 目录各成一个域，identity+contains、in、contains 均命中。

### B2. 在线文档连接器样板（**至少 +1 个新源**，打通在线「一键接入」全链路）
- Feishu Bitable 已是在线多维表样板。v0.5 再加 **1 个主流在线源**，把「向导四步」在**在线**场景端到端跑通
  （含 OAuth/token、分页、限流、`describe_schema` 的 list-fields、`ConnectorFilter` 翻译）。
- **决策点（待拍板）**：样板源选 **腾讯文档智能表格 / 钉钉多维表 / Notion / Airtable** 之一。
  建议 **腾讯文档智能表格**——国内法务画像最高值，且数据模型与 Feishu Bitable 相近，可复用
  children-OR / 富文本扁平化 / 多维表分页的经验，工程风险最低。
- 其余在线源不在 v0.5 官方实现，靠 B3 指南由社区/用户自助加（这正是「official = 1–2」非目标的兑现方式）。

### B3. 连接器作者指南（doc，兑现原 v0.5 contributor surface）
- 「如何写一个连接器并让它出现在添加向导里」：实现 `DataConnector` + 注册 `type` 工厂 +
  `describe_schema` 约定 + 算子翻译 + 富文本/分页/限流注意事项 + 安全约束（只暴露声明字段）。

## 6. 柱三 —— planner 检索增强（适应更宽泛字段与问法）

### C1. 零结果 `identity` 改写（修 `eq` 假空，**最高优先**）
- 在执行层：若**授权成功**但结果 `no_rows`，且该 plan 的过滤命中**单一 identity 字段 + `eq`**，
  则**确定性**改写为 `identity` + `contains` **重试一次**；审计记录「发生了改写」（保留可观测性）。
- **决策**：选执行层确定性改写，**而非** planner 重试循环——因为这是结构性修复（v0.4.8 identity 语义的
  兜底），不应依赖模型随机性；v0.4.9 当时故意未做，v0.5 正式补上并留审计痕。
- *验收 C1*：回放 dogfood 失败问句（`MOON的法务BP是谁` 等）从假空变正确命中。

### C2. catalog 字段语义增强（适应「更宽泛的字段」）
- 让连接器声明可带 **字段描述 / 同义词 / 示例值**，经 `query_catalog` 注入 planner prompt，
  使陌生表的奇怪列名也能被自然语言问法命中。中英别名扩展并入同一机制。
- 承载层：新表 `field_semantics`（schema v23），统一适用 YAML 声明源与 DB 注册源。

### C2.5. 字段语义召回词生成 + 提示词治理（2026-06-13 负责人新增需求）
- **每当读取用户接入数据库的字段后，按字段名先生成一批语义相似的「召回词」，且为语义匹配而非字符变形，
  避免同义词落空。** 把语义扩展**前移到接入期**，查询期仍保持确定、低延迟（不在查询期调模型）。
- 生成：对每个字段（名+显示名+已有别名+域上下文）调既有唯一模型 seam `AIProvider.complete()`（本地模型即可驱动）
  产出召回词（同义词/口语/中英/行业黑话），写入 `field_semantics.synonyms`（`origin=generated`），注入 planner prompt。
- **提示词治理**：召回词集合是可审阅/可编辑/可版本化/入审计的工件——admin 可在接入 review 步或 CLI 增删；
  召回词只是「近义说法 → canonical 字段」的把手，**不放宽任何字段授权**（命中后仍走字段门），永不携带行值。
- 模型不可用 → 降级为空召回词（功能退化但绝不 fail-open 放宽授权）。查询期 embedding/向量召回 = v0.6 RAG 范围。

### C3. 算子补全（适应「更宽泛的问法」）
- 在**所有**连接器（sqlite_demo / feishu / local_file / 新在线样板）落地 `is_empty` /
  `date_before` / `date_after` / `date_between` 的翻译——schema 早已声明、planner 早被允许，缺的是连接器执行。

### C4. 空结果可解释 + 澄清回流
- `no_rows`（区别于授权拒）时，返回结构化提示（可能的相近字段 / identity 候选）而非干瘪空，
  引导用户澄清；候选始终在 record-scope + 字段门之后，绝不泄漏越权值。

### C5. 验收
- 用一个**字段命名风格迥异**的接入源（如某腾讯文档表）跑一组宽问法；dogfood 失败集全绿；
  泄漏红队测试不回退。

## 7. v0.6+ 明确探索项（**不在 v0.5 建设**，仅 tee-up）

- **PDF / Word / Markdown 正文检索**（RAG / chunking / embedding）。
- **研究问题（负责人提出，作为 v0.6 立项前置）**：
  **本产品的「行级 × 字段级」权限模型，能否治理 RAG 向量检索的查询结果？**
  - 难点初判：自由文本块横跨多个授权边界，无法按字段切；需要 chunk 级 ACL / 检索后过滤 /
    防止答案级泄漏；与现「授权作用在 query plan 上」的不变式不直接兼容。
  - v0.6 设计前先做一个最小验证实验：给文本块打 record-scope 标签，检验检索+合成阶段能否守住
    头号不变式（未授权内容不进 LLM 上下文）。结论决定 RAG 是否纳入产品方向、是否改非目标。

## 9. 影响清单

- **schema**：v22 → v23（`data_sources` 表 + 可能的 source-scoped 审计列）。沿用现有可恢复 rebuild migration 模式。
- **threat-model.md**：新增「运行时接入源」威胁行（疏忽/恶意 admin、凭据入库面）；泄漏红队测试覆盖向导接入源。
- **identity-model.md**：补一节「企业 SSO = 前置 OIDC 反代 → 可信头」，并写明 OIDC 自研不在路线图。
- **README / CHANGELOG / 部署文档**：添加数据源向导用法、SSO 反代部署样例（compose）。
- **依赖**：本地文件连接器需 `openpyxl`（XLSX）；其余尽量复用标准库 + 临时 SQLite，维持轻依赖。
