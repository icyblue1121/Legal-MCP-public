# 承诺矩阵：你的数据/问题会不会出内网？

> 配套：[威胁模型](threat-model.md)、[身份模型](identity-model.md)、
> [本地模型路径计划](2026-06-08-local-model-path-v0.3.5-plan.md)。
> 一句话回答部署者最关心的隐私问题——按你的部署方式如实区分，不夸大。

网关一次问答涉及三段数据，各自的"是否出内网"取决于**你**怎么部署：

| 数据段 | 含义 | 默认（云端模型） | 配置本地模型后 |
| --- | --- | --- | --- |
| 原始数据源 | 连接器背后的表格/库的全量记录 | **始终留在源**——网关只按授权读取需要的字段，从不整库搬运 | 同左 |
| 披露字段 | 实际返回给 AI/客户端的、已授权的字段子集 | 会进入所调用模型的上下文 | **留在内网**（模型在内网） |
| 问题文本 | 用户问的那句话本身 | 会发给所调用的模型 | **留在内网**（模型在内网） |

## 关键区分：模型在哪，决定问题与披露字段是否出网

- **网关本身不外发原始数据**：授权唯一门（DB grant，控制台管理）+ 行级 `record_scope`
  保证只有授权字段被披露。这条与模型在哪**无关**，始终成立。
- **问答必然要过一个 LLM**。这个 LLM 是云端还是本地，决定**问题文本 + 披露字段**会不会
  离开内网：
  - 用云端模型（默认 OpenAI 兼容端点）：问题文本与披露字段会进入外部模型上下文，
    可能被其缓存/记录。
  - 用本地/自托管模型（Ollama / vLLM，见
    [本地模型路径](2026-06-08-local-model-path-v0.3.5-plan.md)）：整段问答留在内网。

## 因此可以诚实承诺的话术

- 任何部署："**原始数据不出源，只有授权字段被披露，每次披露都被审计。**"
- **仅当配置本地模型**："**问答全程不出内网。**" —— 不配本地模型时**不要**这样承诺。

如何验证你确实在本地模型上：`legal-mcp doctor --probe-ai` 会探测配置的 AI 端点并回显
其 base URL 与模型名；断网后仍能完成问答即证明无外部 LLM 依赖（见
[本地模型 demo](../../examples/legal-demo/LOCAL-MODEL-DEMO.md)）。
