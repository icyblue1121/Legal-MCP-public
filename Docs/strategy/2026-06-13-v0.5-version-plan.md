# Legal-MCP v0.5 逐版本开发计划

> 战略/为什么：见 `2026-06-13-v0.5-plan.md`（v0.5 路线图）。本文是**可执行的逐版本拆解**——
> 每个「事项」= 一个独立可发布、可单独 tag 的小版本，沿用 v0.4.0–v0.4.9 的细粒度节奏
> （一版一事、各自绿、各自 CHANGELOG/版本号 bump）。
> 立项 2026-06-13。前置 = `v0.4.9`（505 绿）。

## 排序原则

1. **见效快、无新基建的先走**（柱三 planner 增强）——直接修 dogfood 假空、当下就提升体验。
2. **无外部依赖的连接器其次**（本地文件）——给后面的「接入框架」一个可端到端验证的「料」。
3. **接入框架核心再上**（运行时注册表 → 向导 → 凭据/视图）。
4. **在线样板源、文档收尾**，最后 tag `v0.5`。

每个版本默认继承既有不变式：**默认拒、被拒字段取数前剥离、泄漏红队测试不回退、接入/授权变更入审计。**

## v0.5.0 — planner 零结果 `identity` 改写（修 `eq` 假空）

**目标**：彻底修掉 dogfood 反复出现的「授权成功却假空」（如 `MOON的法务BP是谁`）——planner
偶发 emit `name eq <token>`（大小写敏感、单字段猜）而非 `identity contains`。

**范围**
- 在**执行层**（非 planner 重试）做**确定性**改写：一次查询若 `authorize` 通过但结果 `no_rows`，
  且该 plan 的过滤恰为**单一 identity 字段 + `eq`**，则把它改写为 `identity` + `contains`
  **重试一次**；仍空才返回空。
- 改写**入审计**（`agent_runs` / 步骤日志记 `rewrite: eq_to_identity`），保留可观测性——
  不掩盖 planner 真实行为，只在结构上兜底。
- 仅作用于声明了 identity 字段的域；非 identity 字段的 `eq` 不动。

**关键文件 / seam**：`agent_graph.py`（execute 节点后的 no_rows 分支）、`connector_retrieval.py` /
`search_tools.py`（改写并复用 v0.4.8 的 `identity` 展开 + `identity_match` 排序）、`query_plan.py`。

**不做**：不改 planner prompt 行为（那是 v0.5.2/3 的事）；不做多字段笛卡尔重试；只兜「单 identity-eq 假空」这一确定模式。

**验收**：回放 dogfood 失败集（`MOON的法务BP是谁`、`name eq` 各源）从假空变命中；审计能看到改写发生；泄漏红队测试不回退。

## v0.5.1 — 算子补全 `is_empty` / `date_*`

**目标**：`QueryPlan` schema 早已声明、planner 早被允许的 `is_empty` / `date_before` /
`date_after` / `date_between`，在**所有**连接器落地翻译（目前连接器侧未实现 → 用户一问就废）。

**范围**
- `connectors/sqlite_demo.py`、`connectors/feishu_bitable.py`（及后续 `local_file`）各自实现这四个算子的
  原生翻译（SQLite：`IS NULL/''`、`date()` 比较；Feishu：对应 filter 谓词）。
- `ConnectorFilter` 若需新增 value 形态（date range 的二元值），补类型与校验。

**关键文件 / seam**：`connectors/base.py`（`ConnectorFilter` 形态）、各连接器 `_build_filter` / `_where`。

**不做**：不引入新算子（只补已声明未实现的）；不做时区/相对日期解析（「上个月」交给 planner 归一成绝对日期）。

**验收**：每个连接器对四算子有单测；一条「某日期字段 date_between」与「某字段 is_empty」端到端命中。

## v0.5.2 — catalog 字段语义层（description / 示例 / 同义词 → prompt）

**目标**：让陌生表的奇怪列名也能被自然语言命中——给字段挂**语义元数据**并注入 planner prompt。
这是「提示词治理」的**承载层**（v0.5.3 往里填内容）。

**范围**
- 新表 `field_semantics`（schema **v23**）：`source, domain, field, description, examples,
  synonyms(JSON), origin(manual|generated), updated_at`。对 YAML 声明源与未来 DB 注册源统一适用。
- 连接器/catalog 可读取并把字段的 description + synonyms + examples 拼进 planner prompt
  （扩 `query_catalog` 的 `catalog_context_for_prompt`）。
- 同义词并入现有 `field_aliases` 的解析路径（planner 把问句词归一到 canonical field 时一并参考）。

**关键文件 / seam**：`db.py`（建表 + 可恢复 rebuild 迁移）、`query_catalog.py`（catalog 注入）、
`agent_graph.py` `_planner_system_prompt`。

**不做**：不在这一版生成内容（手填或留空都行）；不做查询期 embedding 匹配（语义扩展前移到接入期，见 v0.5.3）。

**验收**：给某字段手填一组 synonyms 后，用同义说法提问能命中该字段；prompt 里能看到注入的语义元数据。

## v0.5.3 — 字段语义召回词生成 + 提示词治理 ⭐（本阶段新增需求）

**目标（用户需求）**：**每当读取用户接入数据库的字段后，按字段名先生成一批语义相似的「召回词」，
并且是语义匹配而非字符变形，避免同义词落空。** 把语义扩展**前移到接入期**，查询期仍保持确定、低延迟。

**机制**
- **触发点**：接入数据源 / catalog 刷新 / 显式重算后，对每个字段（字段名 + 显示名 + 已有别名 + 域上下文）
  调用既有唯一模型 seam `AIProvider.complete()` 生成一批召回词——**同义词、口语说法、中英对照、行业黑话**。
  例：`legal_bp`「法务BP」→ {法务对接人, 法务负责人, 法律BP, 法务接口人, legal partner, 法务 business partner}。
  - 复用 `AIProvider`（[agent_graph.py 的单一模型出口]）→ **本地模型即可驱动**，不引入新依赖，符合
    「最敏感场景模型不出网」的承诺。
- **语义而非字符串**：召回词由模型按语义生成（不是 `legal_bp→legalbp` 这种变形），写入 v0.5.2 的
  `field_semantics.synonyms`，`origin=generated`。查询期 planner 拿到的是这份**先验**，不在查询期再调模型。
- **提示词治理（核心约束）**：生成的召回词集合是**可审阅、可编辑、可版本化、入审计**的工件，
  而非藏在模型权重里的不可见行为：
  - admin 可在接入 review 步（v0.5.7 集成）或 CLI 增删召回词；改动记 `origin=manual` + 审计。
  - 召回词只是「把用户的近义说法映射到 canonical field」的**字段把手**——**不放宽任何字段授权**：
    召回词命中字段后仍走字段门，未授权字段照拒。召回词永不携带行值。
  - 提供 `legal-mcp recall-terms <source>`（生成/重算/导出 review），让接入期的 prompt 内容可治理、可复现。

**关键文件 / seam**：`ai_provider.py`（生成调用）、新 `recall_terms.py`（生成 + 落 `field_semantics`）、
`db.py`、CLI（`recall-terms` 子命令）、v0.5.2 的 prompt 注入路径。

**schema**：复用 v0.5.2 的 `field_semantics`（不再 bump）。

**依赖**：v0.5.2（承载层）。可对**任何**catalog（含现有 YAML 源）headless 运行；完整的「接入即生成 + 人工 review」
UX 在 v0.5.7 向导里闭环。

**不做**：不做查询期 embedding / 向量召回（那是 v0.6 RAG 探索）；不让召回词绕过字段门或行级范围；
模型不可用时**降级为空召回词**（功能退化，但绝不阻断或 fail-open 放宽授权）。

**验收**：对一个字段名古怪的接入源跑生成 → `field_semantics` 出现合理召回词；用召回词里的近义说法提问命中正确字段；
admin 删掉一个错误召回词后即时生效并入审计；未授权字段即便被召回词命中仍被拒（泄漏红队测试覆盖）。

## v0.5.4 — `no_rows` 可解释 + 澄清回流

**目标**：授权通过但空结果时，别给用户干瘪的「找不到」，返回结构化提示引导澄清——且绝不泄漏越权值。

**范围**
- `no_rows`（区别于授权拒）时返回：可能的相近字段、identity 候选（复用 `identity_match` 的候选列表）、
  「换个说法/确认是代号还是名称」的引导。
- 候选始终在 record-scope + 字段门**之后**生成，越权行/字段绝不出现在提示里。

**关键文件 / seam**：`agent_graph.py`（`no_rows` 分支 + `format_answer`）、`identity_match.py`。

**不做**：不猜测放宽过滤自动重查（除 v0.5.0 那一处确定性改写外）；提示只含元数据与已授权候选。

**验收**：空结果问句返回带候选/引导的结构化回答；泄漏红队测试确认提示零越权值。

## v0.5.5 — `local_file` 本地文件连接器（CSV / XLSX / JSON / Markdown frontmatter）

**目标**：第一个「本地结构化/半结构化表格」读直通连接器，给接入框架一个无外部依赖、可端到端测的源。

**范围**
- 新 `connectors/local_file.py` 实现 `DataConnector`：
  - `describe_schema()`：解析表头（CSV/XLSX）、键（JSON/JSONL）、frontmatter 键（MD）得列。
  - `catalog()`：由 review 后声明生成；`query()`：执行 `ConnectorFilter`。
- **半结构化映射**：JSON 数组/JSONL → 行；每记录键 → 列。Markdown frontmatter → 每文件一行、
  frontmatter 键为列；**正文不进**（正文 = v0.6 范围，这里明确切线）。
- **算子下推（决策已定）**：把加载的表落入**临时/内存 SQLite** 复用 `sqlite_demo` 的算子翻译
  （eq/contains/in/or_fields + like/nocase + v0.5.1 的 is_empty/date_*），一次拿到全部算子且与现有路径一致。

**关键文件 / seam**：`connectors/local_file.py`、复用 `sqlite_demo` 算子翻译、`scaffold.py`（草稿）。

**schema**：无（源声明此时仍走 YAML / 或等 v0.5.6 入库）。

**依赖**：`openpyxl`（XLSX 解析）——本版引入的唯一新依赖；CSV/JSON/MD 用标准库 + `PyYAML`（已有）。

**不做**：不解析 PDF/Word/正文；不做文件监听/增量（接入期快照即可，重算靠 catalog 刷新）。

**验收**：一个 XLSX 与一个含 frontmatter 的 `.md` 目录各成一个域，identity+contains / in / date_between 均命中；
`describe_schema` 正确列出列。

## v0.5.6 — 运行时源注册表（`data_sources`，热生效）

**目标**：把「源配置」从静态 YAML、启动期加载，升级为 **DB 持久化、运行时可增**，且**免重启生效**——
接入框架的架构基座。

**范围**
- 新表 `data_sources`（schema **v24**）：`id, name, type, status(draft|active|disabled),
  config_json, secret_ref, created_by_user_id, created_at, updated_at`。`config_json` = 人工 review 后的
  域/字段/identity/record_scope/别名声明（scaffold 草稿的落库形态）。
- `connector_config` 增「从 DB 行构建连接器」路径，与现有「从 YAML 构建」**共用** `build_connector_setup`
  的 `type`→工厂分派；最终 `CompositeConnector` = **YAML 声明源 ∪ DB active 源** 合并。
- **热生效**：`_catalog_for_database` 本就按请求重建 catalog + 读 `data_source_state`，把「DB active 源」
  纳入同一重建路径即可免重启；加一层以源表 `updated_at` 为 key 的失效缓存控成本。

**关键文件 / seam**：`db.py`（建表 + 可恢复 rebuild 迁移）、`connector_config.py`（DB 构建路径）、
`agent_graph.py` `_catalog_for_database`（合并 + 缓存失效）。

**不做**：本版**不出 UI**（仅 DB 驱动构建路径，靠插入 active 行验证）；凭据仍走 env 引用（明文入库留 v0.5.8 决策）。

**验收**：向 `data_sources` 插一行 active（指向已实现的 `local_file` 或 sqlite）→ 下一次 agent 查询的 catalog
立即含该域，无需重启；disable 该行 → 域离开 catalog（`unsupported_domain`）。

## v0.5.7 — 添加数据源向导 + 安全护栏

**目标**：admin 后台四步「一键」接入一个源；向导与安全护栏**同版发布**（无护栏的向导 = 一键越权，不可单独发）。

**范围**
- **向导（server-rendered，沿用 v1.5 admin 设计系统）**，全程**只露字段名/元数据，永不显示行值**：
  1. 选类型（= 已注册连接器工厂；未实现的类型不出现）。
  2. 连接（在线→app_token+凭据引用；本地→上传/路径）。
  3. 自省（`describe_schema()` 列出表与列）。
  4. review + 启用（勾字段、标 identity/sensitive、设 `record_scope`、填别名；**触发 v0.5.3 召回词生成并供 review**）
     → 写 `data_sources(status=active)`。
- **安全护栏（必须）**：仅 `admin` 可加/改/删源；新源默认 `record_scope:none` + **零 grant** = 默认拒；
  只暴露 review 时显式声明的字段；接入/变更/启停均入审计（高价值事件）。
- **泄漏红队测试扩到「向导接入的源」**：over-request / projection / denied-only 对运行时新增源同样 fail-closed。

**关键文件 / seam**：`admin_database.py` / 新 `admin_data_sources.py`、`scaffold.py`（向导内核）、
`recall_terms.py`（review 步集成）、`tests/test_disclosure_leakage.py`（扩用例）。

**不做**：不支持明文凭据入库（留 v0.5.8 决策）；不做在线 OAuth 跳转登录（凭据靠 token/引用）。

**验收**：用一个本地 CSV 走完四步 → 域进 catalog；**授权前**任意非 admin 查它得授权拒绝（非空值）；
接入事件入审计；泄漏测试全绿。

## v0.5.8 — 凭据存储 + Data Sources 视图 CRUD

**目标**：补齐在线源接入所需的凭据存储，并把 Data Sources 视图升级为可增删管理。

**范围**
- **凭据存储（决策点，§10）**：默认 **env 引用**（`secret_ref` 存 env 变量名，明文不入库）；
  **加密入库**作为显式 opt-in——部署设了主密钥 `LEGAL_MCP_SECRET_KEY` 才开放后台贴 token。
  选加密入库则 `threat-model.md` 增补「DB 泄漏 = 秘钥泄漏」威胁行（可能 schema **v25** 加密列）。
- **Data Sources 视图**：现「连/断」基础上加「+ 添加数据源」入口、draft/active/disabled 状态、删除（解绑）源
  （= 注册表移除 + 域离开 catalog）。

**关键文件 / seam**：`admin_data_sources.py`、`db.py`（可选加密列）、`threat-model.md`。

**不做**：不内置密钥轮换/KMS 集成（env 引用即把轮换交给部署侧）。

**验收**：env 引用模式下接入一个需凭据的源成功；（若启用）加密入库往返正确且 DB 里无明文；
后台可删除源并立即从 catalog 消失；接入/删除入审计。

## v0.5.9 — 在线样板连接器（待定源）

**目标**：把「向导四步」在**在线**场景端到端跑通，证明一键接入对线上文档同样成立。

**范围**
- 实现 **1 个主流在线源**连接器（含 OAuth/token、分页、限流、`describe_schema` 的 list-fields、
  `ConnectorFilter` 翻译 eq/contains/in/or_fields + v0.5.1 算子）。
- **决策点（§10，待拍板）**：腾讯文档智能表格（推荐，国内法务最高值且模型近 Feishu）/ 钉钉多维表 / Notion / Airtable。
- 其余在线源**不在 v0.5 官方实现**，靠 v0.5.10 的连接器作者指南由社区/用户自助加（兑现「official = 1–2」非目标）。

**关键文件 / seam**：`connectors/<source>.py` + 其 HTTP client，注册进 `build_connector_setup` 的 `type` 工厂。

**不做**：不一次做四个在线源；不做源侧逐用户委派（连接器始终单一共享服务凭据，见 identity-model §身份穿透）。

**验收**：通过向导接入该在线源 → 自省字段 → review → 查询命中；contains/in/identity 全链路 + 富文本扁平化正确。
