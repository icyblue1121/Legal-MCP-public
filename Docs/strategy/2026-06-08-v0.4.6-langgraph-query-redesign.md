# v0.4.6 plan — LangGraph query redesign and turn-safe conversations

> **Status (2026-06-09): IMPLEMENTED.** Sections A–G shipped. Schema v22
> (`agent_steps.turn_id` + `agent_turn_context`). Regression tests 1–8 live in
> `tests/test_agent_turn_isolation.py`; context/reset units in
> `tests/test_conversation_context.py`. Full suite green apart from one
> pre-existing, unrelated `DROP COLUMN` environment failure in `test_schema.py`.
> Connector-owned fast intents (open choice 3) intentionally deferred.

> North star unchanged: *小问题别来问我，问 AI；AI 只回答你有权知道的那一部分。*
>
> v0.4.6 fixes a query-orchestration bug exposed on 2026-06-08: `agent_query`
> can return `{"projects":[]}` even when the user is authorized and the data
> exists, because a stale `query_plan` from an earlier turn survives in the
> LangGraph checkpoint. The same investigation also showed that the global
> hard-coded fast path is no longer appropriate once domains and fields come from
> arbitrary connectors.

## Problem statement

Two failures were exposed by the same dogfood session, but they have different
reproduction conditions and must be tested separately:

1. **Global fast path over-parses natural language.** `agent_fast_path._project_token()`
   assumes legal-demo phrasing and extracts the text before the first `的`. For
   a question like `请查询项目代码 VT-0001 的法务 BP`, it can build a plan equivalent
   to `name = "请查询项目代码 VT-0001"` instead of `project_code = "VT-0001"`.
   This reproduces on both the LangGraph path and the linear fallback path,
   because both call `classify_question()` and `plan_fast_path()`.
2. **LangGraph state is not turn-local.** `run_agent_query()` passes the user
   `thread_id` as the LangGraph checkpoint thread. A later invocation with the
   same `thread_id` starts with new `question` but may still contain old
   `query_plan`, `tool_result`, `answer`, and planner metadata. Then
   `classify_question()` treats any existing `query_plan` as a pre-supplied plan
   and skips planning. This reproduces only when the LangGraph path is active
   with checkpointing; the linear fallback starts from `{"question": question}`
   each call and does not hydrate old checkpoints.

The combined outcome is a silent false negative: the tool call is marked
successful, but the wrong plan searches zero rows. The v0.4.6 test suite must
therefore force coverage of the LangGraph path (install LangGraph in CI, or mark
LangGraph-only tests as skipped when the dependency is absent) and also cover the
linear fallback for the fast-path regression.

## Version goal

v0.4.6 makes each user turn an isolated query transaction while preserving
multi-turn conversational context as explicit, authorized input. The graph must
never execute a stale plan from a previous user message, and the gateway must not
use a built-in legal-project fast path for arbitrary connector-backed data.

## Design principles

- **One turn, one plan.** Every `agent_query` invocation creates a fresh plan
  from the current question plus explicit conversation context. A plan is valid
  only for the current turn.
- **Conversation memory is input, not mutable graph state.** Prior turns may help
  resolve ellipses like `这个项目的官网呢`, but they are loaded through a dedicated
  context node and copied into the planner prompt. They are not inherited as
  `query_plan` or `tool_result`.
- **The live connector catalog is the source of query semantics.** A generic
  graph may classify access/clarify/search intent, but it must not hard-code
  project-code grammar, field names, or domain-specific business assumptions.
- **Fail closed and legible.** If the planner cannot bind the question to a
  registered domain, field, and filter, return a clarification or structured
  planning error. Do not return an empty result caused by an invalid heuristic.

## Architecture

### A. Remove the global business fast path from normal planning

Keep deterministic handling only for intents that do not depend on source schema:

- `describe_my_access` / permission-scope questions.
- `structured_query` requests where the caller has already supplied a validated
  plan.
- Optional future connector-owned fast intents, but only when the connector
  explicitly declares them and returns a catalog-valid plan.

Remove `plan_fast_path()` from the default natural-language search path for
project fields and license fields. The default planner should always use the
live `QueryCatalog` built from the connector setup.

**Files likely touched:**

- `src/legal_mcp/agent_fast_path.py`
- `src/legal_mcp/agent_graph.py`
- `tests/test_agent_fast_path.py`
- `tests/test_agent_graph.py`

**Acceptance:**

- A verbose question like `请查询项目代码 VT-0001 的法务 BP 是谁` no longer uses the
  global fast path or produces `name = "请查询项目代码 VT-0001"`.
- Access-scope questions still bypass the model and call `describe_my_access`.
- Existing tests that depend on deterministic field planning are rewritten to
  use `StubAIProvider`, because schema-bound planning is now the planner's job.

### B. Introduce explicit turn identity separate from conversation identity

Rename the mental model:

- `conversation_id`: the user-visible thread id supplied by the client and
  returned to the client. Used to retrieve prior turn summaries.
- `turn_id`: a new UUID generated for each `agent_query` invocation.
- `checkpoint_thread_id`: internal LangGraph checkpoint key, derived from
  `turn_id`, not from `conversation_id`.

`run_agent_query(thread_id=...)` should keep returning the supplied `thread_id`,
but `graph.invoke(..., config={"configurable": {"thread_id": checkpoint_thread_id}})`
must use a fresh per-turn id. This preserves observability without allowing one
turn's state to become the next turn's state.

**Files likely touched:**

- `src/legal_mcp/agent_graph.py`
- `src/legal_mcp/agent_observability.py`
- `tests/test_agent_graph.py`

**Acceptance:**

- Two calls with the same client `thread_id` produce two distinct internal
  checkpoint ids.
- The second call's state cannot contain the first call's `query_plan`,
  `tool_result`, `answer`, `error`, `query_type`, `model_intent`,
  `clarify_reason`, `planner_source`, or `planner_reason` unless a node
  explicitly writes them for that turn.
- `agent_runs.thread_id` remains the client conversation id for operator audit.
- Langfuse mapping is explicit: `conversation_id` becomes the session id that
  groups turns, while `turn_id` becomes the trace id/name suffix for one query
  transaction.

### C. Make turn-local state reset explicit

Even with per-turn checkpoints, add a defensive state boundary at graph entry.
This is not a substitute for per-turn checkpoints; it is a second guardrail for
callers, tests, and future refactors. The initial state for natural-language
`agent_query` should include only:

```python
{
    "conversation_id": conversation_id,
    "turn_id": turn_id,
    "question": question,
    "input_mode": "natural_language",
}
```

For `structured_query`, include:

```python
{
    "conversation_id": conversation_id,
    "turn_id": turn_id,
    "question": "structured_query",
    "input_mode": "structured",
    "query_plan": plan,
}
```

Then replace this current guard:

```python
if state.get("query_plan") is not None:
    return {"query_type": "search"}
```

with a guard based on the explicit input mode:

```python
if state.get("input_mode") == "structured":
    return {"query_type": "search"}
```

Natural-language turns must always run classification/planning, regardless of
whether a checkpoint or caller accidentally contains a `query_plan`.

Do not implement this as "clear only `query_plan`." The whole turn-local set must
be absent at the start of a natural-language turn:

```python
TURN_LOCAL_KEYS = {
    "query_type",
    "normalized_question",
    "query_plan",
    "tool_result",
    "answer",
    "error",
    "tool_calls",
    "model_intent",
    "clarify_reason",
    "planner_source",
    "planner_reason",
}
```

If the LangGraph checkpoint id is fresh, these keys will be absent naturally. If
a future graph entry point accepts an existing state object, `start_turn` must
drop these keys before running planning.

**Files likely touched:**

- `src/legal_mcp/agent_graph.py`
- `tests/test_agent_graph.py`

**Acceptance:**

- A natural-language call with a stale `query_plan` injected into state still
  replans from `question`.
- A natural-language call with stale `error`, `tool_result`, or `answer`
  injected into state does not short-circuit or return the old result.
- A structured query still executes the caller-supplied plan.

### D. Add a conversation-context node

Add a graph node before planning:

```text
load_conversation_context -> classify_question -> build_query_plan -> ...
```

This node reads previous successful turns for the same `conversation_id` and
returns a narrow, structured context object. It must not hydrate the full prior
LangGraph state. Suggested shape:

```python
{
    "conversation_context": {
        "recent_entities": [
            {"domain": "project", "identity": {"project_code": "VT-0010", "name": "虚拟测试项目010-山海计划"}}
        ],
        "recent_fields": ["website", "legal_bp"],
        "last_successful_tool": "project/search",
    }
}
```

The context resolver should store only information already returned to the user
or identity fields that the authorization model treats as safe. It should not
store unauthorized field values or raw connector rows.

Minimal implementation option: derive context from prior successful
`agent_steps.plan_json` plus the corresponding safe answer/result. The plan alone
can support "same filter, new field" follow-ups, but it is not enough when the
previous turn filtered by one identity and the follow-up needs another. Example:
if the prior plan filtered by `name`, the resolved `project_code` must come from
the authorized result, not from `plan_json`. Document this boundary in tests so
the implementation does not overclaim context support.

The planner prompt can then resolve follow-ups:

- `这个项目的官网是什么` uses the latest project identity from context.
- `那它的法务 BP 呢` uses the latest project identity but requests `legal_bp`.
- If multiple candidate entities exist, return a clarification.

**Files likely touched:**

- `src/legal_mcp/agent_graph.py`
- `src/legal_mcp/agent_steps.py` or a new `src/legal_mcp/conversation_context.py`
- `src/legal_mcp/schema.sql` if a dedicated context table is added
- `tests/test_agent_graph.py`
- `tests/test_schema.py`

**Acceptance:**

- Follow-up questions can use prior entity context without reusing prior plans.
- If the previous turn failed or returned no rows, it is not promoted as the
  current entity context.
- Ambiguous context produces `clarify_query`, not a guessed query.

### E. Redesign the LangGraph nodes around a typed query transaction

The graph should make the turn lifecycle explicit:

```text
start_turn
  -> load_conversation_context
  -> classify_intent
  -> plan_query
  -> validate_plan
  -> authorize_plan
  -> execute_plan
  -> format_answer
  -> record_turn_context
```

The existing `normalize_query` node can be folded into `start_turn` or
`plan_query`; the current placeholder `build_query_plan` node should be removed
or replaced by the real `plan_query` node. Do not leave a topology-only node that
suggests planning happens somewhere it does not.

Node responsibilities:

- `start_turn`: assigns `turn_id`, creates a clean state envelope, records
  conversation id.
- `load_conversation_context`: loads safe prior context only.
- `classify_intent`: returns `access`, `search`, `clarify`, or `structured`.
  It may be deterministic for access; otherwise use the planner/model.
- `plan_query`: produces exactly one `QueryPlan` or a clarification reason.
- `validate_plan`: checks the plan against the live connector catalog.
- `authorize_plan`: checks field and row authorization.
- `execute_plan`: reads from SQLite or connector, as today.
- `format_answer`: serializes a tool result or error.
- `record_turn_context`: writes safe context from successful results for future
  turns.

This structure separates the two things that are currently conflated in
`classify_question()`: deciding what kind of request this is, and producing the
actual query plan.

**Files likely touched:**

- `src/legal_mcp/agent_graph.py`
- `src/legal_mcp/agent_router.py`
- `tests/test_agent_graph.py`
- `tests/test_agent_steps.py`

**Acceptance:**

- The graph has no node where the presence of an old `query_plan` changes a
  natural-language turn's control flow.
- Planning failures are visible as `clarify` or `error`, not successful empty
  result sets.
- Agent step audit records both the planner source and the exact plan selected
  for the current `turn_id`.

### F. Fix multi-turn planner audit

`agent_steps` currently uses `unique(thread_id, step_index)`, while each turn
starts planner attempts at `step_index = 1`. A second turn in the same
conversation can collide, and `_record_step_safely()` swallows the SQLite error.
That makes per-turn plan audit silently disappear exactly when it is needed to
debug stale plans.

Migrate planner telemetry to use explicit turn identity:

- Add `conversation_id` and `turn_id` to `agent_steps`, or reinterpret existing
  `thread_id` as `conversation_id` and add a non-null `turn_id`.
- Replace `unique(thread_id, step_index)` with
  `unique(conversation_id, turn_id, step_index)`.
- Update `list_agent_steps()` to support listing by conversation and optionally
  by turn.
- Update `_record_step_safely()` so integrity errors are not silently swallowed
  in tests. Production can still fail closed for telemetry, but tests must catch
  collisions.
- Extend the `planner_source` check constraint if new planner sources are added,
  for example `context`, `connector_fast_intent`, or `structured`.

**Files likely touched:**

- `src/legal_mcp/schema.sql`
- `src/legal_mcp/agent_steps.py`
- `src/legal_mcp/agent_graph.py`
- `tests/test_agent_steps.py`
- `tests/test_schema.py`
- `tests/test_agent_graph.py`

**Acceptance:**

- Two turns with the same `conversation_id` and `step_index = 1` both persist
  their selected plan under distinct `turn_id`s.
- Test failures surface if planner telemetry collides.
- Operator audit can show the exact current-turn plan that produced an empty
  result.

### G. Improve result semantics for empty results

An authorized zero-row result is still possible, but it should be diagnosable.
When a query returns no rows, include the executed plan in server-side audit and
optionally a safe client message:

```json
{
  "projects": [],
  "diagnostic": {
    "reason": "no_rows",
    "matched_authorization": true
  }
}
```

The client-safe `agent_query` response should still avoid leaking excessive
plans by default, but internal audit must preserve the current-turn plan so an
operator can distinguish:

- no matching data,
- no visible rows,
- denied field/filter,
- planner could not ground the user's phrase.

Client-safe nuance: `tools._client_safe_agent_result()` currently drops the raw
`result` object and plan details, forwarding user-visible data through the
serialized `answer` string and a sanitized `tool_calls` list. Any diagnostic
intended for the client must therefore be included deliberately in the formatted
answer; any diagnostic intended only for operators should live in `agent_steps`,
`agent_runs`, or audit details.

**Files likely touched:**

- `src/legal_mcp/search_tools.py`
- `src/legal_mcp/connector_retrieval.py`
- `src/legal_mcp/tools.py`
- `src/legal_mcp/agent_graph.py`
- `tests/test_search_tools.py`
- `tests/test_connector_retrieval.py`

**Acceptance:**

- A wrong filter no longer looks the same as a permission denial in audit.
- Client-safe output does not expose unauthorized field names or values.

## Regression tests required before tagging v0.4.6

1. **No stale plan across same conversation id.**
   - Force the LangGraph path. If LangGraph is not installed, the stale-checkpoint
     test must skip explicitly instead of passing through the linear fallback.
   - Call `run_agent_query(..., thread_id="same", question="请查询项目代码 VT-0001 的法务 BP 是谁")`
     with a stub model returning `project_code=VT-0001`, `return_fields=["legal_bp"]`.
   - Call `run_agent_query(..., thread_id="same", question="VT-0010 的官网是什么")`
     with a stub model returning `project_code=VT-0010`, `return_fields=["website"]`.
   - Assert the second answer contains `website`, not `legal_bp`, and the stored
     plan filters `VT-0010`.

2. **Access question does not poison next turn.**
   - First turn asks `我有什么权限`.
   - Second turn asks `VT-0010 的官网是什么`.
   - Assert the second turn plans and executes a search, not another access path.

3. **Failed turn does not become entity context.**
   - First turn asks a query that returns no rows.
   - Second turn asks `它的官网是什么`.
   - Assert clarification, not a guessed stale plan.

4. **Follow-up uses context but replans.**
   - First turn asks `VT-0010 的项目名称是什么`.
   - Second turn asks `它的官网是什么`.
   - Assert the second turn's plan is newly generated with `return_fields=["website"]`
     and the prior entity identity only appears as context.

5. **Global business fast path disabled.**
   - Assert project-field and license-field natural-language queries do not call
     `plan_fast_path()` or do not produce a fast-path plan.
   - Assert access-scope fast path still works.

6. **Connector-backed arbitrary domain works the same way.**
   - Configure a fake connector domain with non-project fields.
   - Ask two same-thread questions against different fields.
   - Assert no stale plan and no project-specific parsing.

7. **Linear fallback still catches fast-path removal.**
   - Run without LangGraph or force `_run_linear_graph`.
   - Ask `请查询项目代码 VT-0001 的法务 BP 是谁`.
   - Assert the query is model-planned or clarified; it must not build
     `name = "请查询项目代码 VT-0001"`.

8. **Planner audit is turn-keyed.**
   - Run two successful turns with the same `conversation_id`.
   - Assert both have a selected step with `step_index = 1` and distinct
     `turn_id`s.
   - Assert no SQLite integrity error is swallowed in the test path.

## Migration and compatibility

- Existing clients can keep passing `thread_id`. It remains the conversation id.
- Internal checkpoint rows become per-turn. Operators should use `agent_runs`
  and agent-step audit to inspect conversations across turns.
- `agent_steps` requires a schema migration. Existing rows can be backfilled with
  `conversation_id = thread_id` and `turn_id = 'legacy-' || id` or left readable
  through a compatibility query; new rows must always carry a real turn id.
- Existing checkpoints created before v0.4.6 should not be replayed into new
  natural-language turns. The safest migration is to start using fresh
  `checkpoint_thread_id`s immediately and leave old checkpoint rows as historical
  debug data.
- Keep `structured_query` behavior stable: supplied structured plans still skip
  natural-language planning, but only when `input_mode == "structured"`.

## Non-goals

- No new business-domain parser for project codes.
- No attempt to infer field aliases from source data values.
- No storage of full connector rows as conversation memory.
- No weakening of field-level or row-level authorization to make follow-ups work.

## Release gates

- All existing authorization and disclosure leakage tests pass.
- New v0.4.6 regression tests pass with both LangGraph and linear fallback paths.
- Manual dogfood:
  1. Ask `我有什么权限`.
  2. Ask `VT-0001 的法务 BP 是谁`.
  3. Ask `VT-0010 的官网是什么`.
  4. Ask `它的法务 BP 呢`.
  5. Confirm each turn's server-side plan is current-turn correct.

## Open design choices

1. **Where to store conversation context.** Minimal option: derive it from
   `agent_runs` plus current-turn successful result. More robust option: add a
   small `agent_turn_context` table with `conversation_id`, `turn_id`,
   `safe_entities_json`, and `created_at`.
2. **How much plan detail to return to clients.** Keep current client-safe
   behavior by default, but consider an admin/debug flag that returns sanitized
   plan details.
3. **Whether connector-owned fast intents are worth v0.4.6 scope.** They are
   safe architecturally if they are connector-local and catalog-validated, but
   v0.4.6 can ship without them.
